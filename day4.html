What is JavaScript?
The only language that runs natively in every browser

JavaScript's Role in Web Development

JavaScript (JS) is a high-level, interpreted, dynamically-typed programming language. It is the only language that runs natively inside a web browser, making it the language of the web. It was created in 1995 by Brendan Eich in just 10 days — which explains some of its quirky behaviour.

In the full stack picture: HTML is the structure (skeleton), CSS is the style (skin), and JavaScript is the behaviour (muscles and brain). Every interaction you see on a website — form validation, dropdown menus, infinite scroll, toast notifications, live search — is JavaScript.

JavaScript Engines: V8 (Chrome, Node.js) · SpiderMonkey (Firefox) · JavaScriptCore (Safari). The V8 engine compiles JS to machine code using JIT (Just-In-Time) compilation, making it far faster than early JS interpreters.

JavaScript Can Run In Two Places

In the Browser (Frontend): Manipulates the DOM, handles events, makes API calls, animates elements. Runs sandboxed — cannot access the file system for security.

In Node.js (Backend/Server): Runs on a server, can read/write files, connect to databases, handle HTTP requests. This is what makes JavaScript "full stack" — one language for both client and server.

JavaScript — How to Add JS to HTML
/* ── 1. External (BEST — separation of concerns) */
/* In HTML: */
<script src="script.js" defer></script>
/* defer: downloads in background, runs AFTER HTML is fully parsed */
<script src="script.js" async></script>
/* async: downloads in background, runs AS SOON as downloaded (order not guaranteed) */

  console.log('Hello from internal script');
</script>

/* ── 3. Inline (avoid) */
<button onclick="alert('clicked')">Click</button>

console.warn('warning');    // Yellow warning
console.error('error');    // Red error
console.table([1,2,3]);    // Table view
console.dir(document);     // Object tree view

The Three Keywords

JavaScript — var vs let vs const
// ── VAR: function-scoped, hoisted, avoid in modern JS ──
var name = 'Poorv';
var name = 'Someone';  // OK — redeclaration allowed (bug-prone!)
name = 'Another';      // OK — reassignment allowed

// var is function-scoped, NOT block-scoped
if (true) {
  var x = 10;  // escapes the if-block!
}
console.log(x); // 10 — accessible outside! (var leaks)

// ── LET: block-scoped, modern standard for mutable values ──
let age = 21;
age = 22;        // OK — reassignment allowed
// let age = 23; // ❌ Error — cannot redeclare

if (true) {
  let y = 20;   // stays inside this block
}
// console.log(y); // ❌ ReferenceError — y not defined here

// ── CONST: block-scoped, cannot be reassigned ──
const PI = 3.14159;
// PI = 3;  // ❌ TypeError — Assignment to constant variable
// const PI = 3.14; // ❌ Error — cannot redeclare

// IMPORTANT: const with objects/arrays — the REFERENCE is const,
// but the contents CAN be mutated!
const person = { name: 'Poorv', age: 21 };
person.age = 22;       // ✅ OK — mutating property
person.city = 'Junagadh'; // ✅ OK — adding property
// person = {};        // ❌ Error — reassigning the reference

const nums = [1, 2, 3];
nums.push(4);           // ✅ OK — mutating the array
// nums = [5, 6];      // ❌ Error — reassigning the reference

// ── BEST PRACTICE ──
// 1. Always use const by default
// 2. Use let only when you need to reassign
// 3. Never use var in modern code

Data Types
7 primitive types + Objects — what every value in JS is

Primitive vs Reference Types

JavaScript has two categories of types. Primitives are stored directly in the variable — copying a primitive copies the actual value. Reference types (Objects, Arrays, Functions) store a pointer (memory address) — copying a reference type copies the pointer, so both variables point to the same object in memory.

There are 7 primitive types: String, Number, Boolean, null, undefined, Symbol, BigInt.
Everything else is an Object — Arrays, Functions, Dates, RegExp, etc. — which means typeof [] returns "object" and typeof function(){} returns "function".

Operators
Arithmetic, comparison, logical, and modern ES6+ operators
JavaScript — All Operators
// ── ARITHMETIC ──
10 + 3   // 13  addition
10 - 3   // 7   subtraction
10 * 3   // 30  multiplication
10 / 3   // 3.333... division
10 % 3   // 1   modulus (remainder)
2 ** 10  // 1024 exponentiation (ES7)

// ── ASSIGNMENT ──
let x = 10;
x += 5;   // x = x + 5  → 15
x -= 3;   // x = x - 3  → 12
x *= 2;   // x = x * 2  → 24
x /= 4;   // x = x / 4  → 6
x %= 4;   // x = x % 4  → 2
x++;      // post-increment: returns x, THEN adds 1
++x;      // pre-increment: adds 1 THEN returns x
x--;      // post-decrement
--x;      // pre-decrement

// ── COMPARISON ──
5 ==  '5'  // true  — loose equality (type coercion)
5 === '5'  // false — strict equality (no coercion) ← ALWAYS USE THIS
5 !=  '5'  // false — loose inequality
5 !== '5'  // true  — strict inequality ← ALWAYS USE THIS
5 >   3    // true
5 <   3    // false
5 >=  5    // true
5 <=  4    // false

// ── LOGICAL ──
true  && true   // true  — AND: both must be true
true  || false  // true  — OR: at least one must be true
!true           // false — NOT
!!'hello'       // true  — double-NOT: converts to boolean

// Short-circuit evaluation
false && expensiveCall() // expensiveCall never runs
true  || expensiveCall() // expensiveCall never runs

// ── MODERN ES6+ OPERATORS ──

// Nullish Coalescing (??) — use right side if LEFT is null/undefined
const username = null;
const display = username ?? 'Guest';  // 'Guest'
const count = 0;
const total = count ?? 10;     // 0 (0 is NOT null/undefined!)
const total2 = count || 10;    // 10 (|| treats 0 as falsy — often wrong!)

// Optional Chaining (?.) — safely access nested properties
const user = { address: { city: 'Junagadh' } };
const city1 = user?.address?.city;  // 'Junagadh'
const zip   = user?.address?.zip;   // undefined (no error!)
const pin   = user?.contact?.phone; // undefined (contact doesn't exist)
// Without ?.: user.contact.phone → TypeError: Cannot read property 'phone' of undefined

// Ternary operator: condition ? valueIfTrue : valueIfFalse
const age = 20;
const status = age >= 18 ? 'Adult' : 'Minor';  // 'Adult'

// Logical Assignment (ES2021)
let a = null;
a ??= 'default';  // assign 'default' only if a is null/undefined
let b = 0;
b ||= 42;         // assign 42 only if b is falsy
b &&= 100;        // assign 100 only if b is truthy

// typeof
typeof 'hello'     // "string"
typeof 42         // "number"
typeof true       // "boolean"
typeof undefined  // "undefined"
typeof null       // "object" ← famous JS bug
typeof []         // "object" (arrays are objects)
typeof {}         // "object"
typeof function(){} // "function"

// Better type checks
Array.isArray([])         // true ← best for arrays
b instanceof Date         // true if b is a Date
Object.prototype.toString.call(value) // most reliable

Type Coercion & Truthy/Falsy
JavaScript's automatic type conversion — the source of most bugs

What is Type Coercion?

JavaScript is a dynamically-typed, weakly-typed language. It automatically converts values from one type to another when needed. This is called implicit type coercion and it's the source of many JS quirks and bugs. Knowing these rules is critical for interviews.

JavaScript — Coercion, Truthy & Falsy
// ── FALSY VALUES: these evaluate to false in a boolean context ──
// There are EXACTLY 6 falsy values in JavaScript:
false
0          // zero
-0         // negative zero
''         // empty string
null
undefined
NaN

// EVERYTHING ELSE is truthy, including:
'0'        // truthy! (non-empty string)
'false'    // truthy! (non-empty string)
0n         // falsy (BigInt zero)
[]         // truthy! (empty array — common mistake)
{}         // truthy! (empty object)
function(){} // truthy
-1         // truthy (any non-zero number)

// Practical usage
const name = '';
if (name) {
  console.log('has name'); // won't run — '' is falsy
}

// ── STRING + NUMBER COERCION ──
'5' + 3         // '53'  — number converts to string (concatenation)
'5' - 3         // 2     — string converts to number
'5' * '2'        // 10    — both convert to numbers
'10' / '2'      // 5     — both convert to numbers
true + 1         // 2     — true → 1
false + 1        // 1     — false → 0
null + 1         // 1     — null → 0
undefined + 1    // NaN   — undefined → NaN
[] + []           // ''    — both arrays → ''
[] + {}           // '[object Object]'
{} + []           // 0     — {} treated as empty block!

// ── LOOSE (==) vs STRICT (===) EQUALITY ──
1   ==  '1'    // true  — string '1' converts to number 1
0   ==  false // true  — false converts to 0
''  ==  false // true  — both convert to 0
null==  undefined // true  — special case
null==  false // false — null only == null or undefined

// ALWAYS use === and !== to avoid coercion bugs
1   === '1'    // false (different types)
0   === false // false (different types)
⚠️ Classic Interview Trick:
console.log([] == false) → true
console.log([] == ![]) → true (both sides coerce to 0)
These exist because of loose equality's type coercion rules. This is exactly why you should always use ===.

Loops
for, while, for...of, for...in — and when to use each
JavaScript — All Loop Types
// ── FOR LOOP — when you know the count ──
for (let i = 0; i < 5; i++) {
  console.log(i); // 0 1 2 3 4
}

// Loop backwards
for (let i = 4; i >= 0; i--) {
  console.log(i); // 4 3 2 1 0
}

// Loop over array with index
const fruits = ['apple', 'banana', 'mango'];
for (let i = 0; i < fruits.length; i++) {
  console.log(i, fruits[i]); // 0 'apple', 1 'banana', 2 'mango'
}

// ── FOR...OF — iterating over values (arrays, strings, sets) ──
for (const fruit of fruits) {
  console.log(fruit); // 'apple' 'banana' 'mango'
}

// With destructuring + entries()
for (const [index, fruit] of fruits.entries()) {
  console.log(index, fruit); // 0 'apple', 1 'banana', ...
}

// ── FOR...IN — iterating over KEYS of an object ──
const person = { name: 'Poorv', age: 21, city: 'Junagadh' };
for (const key in person) {
  console.log(key, person[key]); // name Poorv | age 21 | city Junagadh
}
// ⚠️ Don't use for...in on arrays — use for...of

// ── WHILE — when condition is checked before each iteration ──
let count = 0;
while (count < 5) {
  console.log(count);
  count++;
}

// ── DO...WHILE — runs AT LEAST once before checking condition ──
let input;
do {
  input = prompt('Enter a number:');
} while (isNaN(input)); // keep asking until valid number

// ── BREAK & CONTINUE ──
for (let i = 0; i < 10; i++) {
  if (i === 3) continue; // skip 3
  if (i === 7) break;    // stop at 7
  console.log(i);          // 0 1 2 4 5 6
}

Functions
Declaration, expression, arrow functions, closures — every form you need
JavaScript — Functions Complete
// ── FUNCTION DECLARATION — hoisted, can call before definition ──
greet('Poorv'); // ✅ works — declaration is hoisted

function greet(name) {
  return `Hello, ${name}!`;
}

// ── FUNCTION EXPRESSION — NOT hoisted ──
const add = function(a, b) {
  return a + b;
};
// add(1, 2) before this line → TypeError: add is not a function

// ── ARROW FUNCTION (ES6) — shorter syntax, lexical 'this' ──
const multiply = (a, b) => a * b;      // implicit return
const square   = n => n * n;           // single param: no parens needed
const sayHi    = () => 'Hi!';          // no params: parens required

// Multi-line arrow function needs braces + explicit return
const divide = (a, b) => {
  if (b === 0) return 'Error: divide by zero';
  return a / b;
};

// Return an object literal — wrap in parens to avoid {} confusion
const makeUser = (name, age) => ({ name, age });

// ── PARAMETERS ──
// Default parameters
function welcome(name = 'Guest', role = 'User') {
  return `Welcome, ${name} (${role})!`;
}
welcome();              // 'Welcome, Guest (User)!'
welcome('Poorv');      // 'Welcome, Poorv (User)!'
welcome('Poorv', 'Admin'); // 'Welcome, Poorv (Admin)!'

// Rest parameters — collects remaining args into an array
function sum(...numbers) {
  return numbers.reduce((acc, n) => acc + n, 0);
}
sum(1, 2, 3);      // 6
sum(1, 2, 3, 4, 5); // 15

// Spread operator — expands an array/object
const nums = [1, 2, 3];
console.log(Math.max(...nums)); // 3 (spread as individual args)
const more = [...nums, 4, 5];  // [1, 2, 3, 4, 5] (copy + extend)
const copy = [...nums];         // shallow copy of array

// ── CLOSURE — function remembers its outer scope ──
function makeCounter() {
  let count = 0;          // private variable — not accessible outside
  return {
    increment: () => ++count,
    decrement: () => --count,
    getCount:  () => count
  };
}
const counter = makeCounter();
counter.increment(); // 1
counter.increment(); // 2
counter.decrement(); // 1
counter.getCount();  // 1
// count is NOT accessible from outside — data privacy via closure

// ── IIFE — Immediately Invoked Function Expression ──
(function() {
  const secret = 'cannot access outside';
  console.log('Runs immediately!');
})();

// ── HIGHER ORDER FUNCTIONS — functions that take/return functions ──
function apply(fn, value) {
  return fn(value);
}
apply(x => x * 2, 5); // 10
apply(Math.sqrt, 16); // 4

// ── RECURSION ──
function factorial(n) {
  if (n <= 1) return 1;   // base case — stops recursion
  return n * factorial(n - 1); // recursive call
}
factorial(5); // 120 (5 * 4 * 3 * 2 * 1)

Scope & Hoisting
Where variables live — the most tricky JavaScript concept
JavaScript — Scope & Hoisting
// ── SCOPE: where a variable is accessible ──

// Global scope: accessible everywhere
const appName = 'MyApp';  // global

function outer() {
  const outerVar = 'I am outer';  // function scope

  function inner() {
    const innerVar = 'I am inner';  // function scope
    console.log(outerVar);   // ✅ inner can see outer (scope chain)
    console.log(appName);    // ✅ inner can see global
  }

  inner();
  // console.log(innerVar); // ❌ outer cannot see inner
}

// Block scope with let/const
{
  let blockVar = 'block';
  console.log(blockVar); // ✅ inside block
}
// console.log(blockVar); // ❌ ReferenceError

// ── HOISTING: declarations are moved to the top of their scope ──

// Function declarations: FULLY hoisted (name + body)
sayHello(); // ✅ 'Hello!' — works before declaration
function sayHello() { console.log('Hello!'); }

// var: hoisted as undefined
console.log(x); // undefined (not an error — hoisted, but value not yet set)
var x = 5;
console.log(x); // 5

// let / const: hoisted but in TEMPORAL DEAD ZONE (TDZ)
// Accessing them before declaration throws ReferenceError
// console.log(y); // ❌ ReferenceError: Cannot access 'y' before initialization
let y = 10;

// Function expressions: only the variable is hoisted, not the function
// greet(); // ❌ TypeError: greet is not a function
const greet = () => 'Hi';

// ── SCOPE CHAIN: JS looks up the chain to find variables ──
const globalVal = 'global';

function level1() {
  const val1 = 'level1';
  function level2() {
    const val2 = 'level2';
    function level3() {
      // Can access: val2, val1, globalVal
      // JS walks UP the chain until it finds the variable or reaches global
      console.log(val2, val1, globalVal);
    }
    level3();
  }
  level2();
}
Temporal Dead Zone (TDZ): Between the start of the block and the let/const declaration, the variable exists (it's hoisted) but cannot be accessed. This region is the TDZ. Accessing a variable in the TDZ throws a ReferenceError. This is why let/const are safer than var — they fail loudly instead of silently returning undefined.

DOM Manipulation
Reading and changing the page with JavaScript

What is the DOM?

The Document Object Model (DOM) is the browser's live, in-memory tree representation of your HTML. Every HTML element becomes a node in this tree. JavaScript can read and modify the DOM — changing text, styles, attributes, adding/removing elements — and the browser instantly re-renders the changes.

JavaScript — Complete DOM Manipulation
// ── SELECTING ELEMENTS ──
const el1  = document.getElementById('hero');            // by ID (returns element or null)
const el2  = document.querySelector('.card');           // first match (any CSS selector)
const el3  = document.querySelector('#hero .title');    // nested selector
const all  = document.querySelectorAll('.card');        // ALL matches (NodeList)
const tags = document.getElementsByTagName('p');       // by tag (HTMLCollection)

heading.innerText               // like textContent but respects CSS visibility

// ── ADDING EVENT LISTENERS (PREFERRED method) ──
const btn = document.querySelector('#myBtn');

// addEventListener(event, callback, useCapture)
btn.addEventListener('click', function(event) {

btn.addEventListener('click', (e) => {
  console.log('Clicked!', e.target); // e.target = element clicked
});

function handleClick(e) { console.log('clicked'); }
btn.addEventListener('click', handleClick);

'dblclick'     // double click

'submit'       // form submitted
'input'        // any value change (real-time)
'change'       // value changed + element blurred
'focus'        // element focused
'blur'         // element lost focus

// Window/Document events
'load'         // page fully loaded (including images)

'resize'       // window resized

// ── THE EVENT OBJECT ──
btn.addEventListener('click', (e) => {
  e.target           // element that triggered the event
  e.currentTarget    // element listener is attached to
  e.type             // 'click'
  e.clientX          // mouse X relative to viewport
  e.clientY          // mouse Y relative to viewport
  e.key              // key pressed ('Enter', 'Escape', etc.)
  e.code             // physical key ('KeyA', 'Space')
  e.ctrlKey          // true if Ctrl was held
  e.shiftKey         // true if Shift was held
  e.preventDefault() // stop default behaviour (e.g. form submit)
  e.stopPropagation() // stop event bubbling up
});

// Events bubble UP the DOM tree by default
//  <body>
//    <div onclick="...">    ← fires 3rd (bubbling up)
//      <button>Click</button>  ← fires 1st
//    </div>
//  </body>

// Stop event from bubbling to parent
btn.addEventListener('click', (e) => {
  e.stopPropagation();  // event stays here
});

// Instead of adding listener to each button:
const todoList = document.querySelector('#todo-list');

todoList.addEventListener('click', (e) => {
  if (e.target.classList.contains('delete-btn')) {
    e.target.closest('li').remove(); // remove the parent li
  }

    e.target.classList.toggle('done');
  }

const form = document.querySelector('#loginForm');

Placement Interview Q&A
Top JavaScript basics questions from real company interviews
  • Q1. What is the difference between var, let, and const?
    var is function-scoped, hoisted as undefined, can be redeclared and reassigned — it's the old way and causes bugs because it leaks out of blocks. let is block-scoped, hoisted but in TDZ (so accessing before declaration throws ReferenceError), can be reassigned but not redeclared. const is block-scoped, cannot be reassigned or redeclared — but for objects and arrays, the contents can still be mutated. Best practice: always use const by default; use let only when reassignment is needed; never use var.
  • Q2. What is the difference between == and ===?
    == is loose equality — it performs type coercion before comparing. So 5 == '5' is true because '5' is converted to 5. === is strict equality — it checks both value AND type without coercion. 5 === '5' is false because one is a Number and the other is a String. Always use === in real code to avoid unexpected coercion bugs. The same applies to != vs !==.
  • Q3. What is a closure in JavaScript?
    A closure is when a function remembers and can access variables from its outer scope even after that outer function has finished executing. This works because functions carry a reference to their outer scope (the "closure"). Closures are used for data privacy (variables that can't be accessed from outside), factory functions, and callbacks. Example: a counter function that returns increment/decrement/getCount — the count variable is private, only accessible through the returned methods.
  • Q4. What is hoisting?
    Hoisting is JavaScript's default behaviour of moving declarations to the top of their scope before code execution. Function declarations are fully hoisted — you can call a function before its definition. var declarations are hoisted as undefined — accessing before declaration gives undefined, not an error. let and const are hoisted but placed in a Temporal Dead Zone — accessing them before declaration throws a ReferenceError. Function expressions and arrow functions are NOT hoisted (only the variable name is, as undefined or TDZ).
  • Q5. What are truthy and falsy values in JavaScript?
    In JavaScript, every value is either truthy or falsy when used in a boolean context (like an if condition). The ONLY falsy values are: false, 0, -0, '' (empty string), null, undefined, and NaN. Everything else is truthy — including '0', 'false', empty arrays [], empty objects {}, and -1. Common mistake: [] == false is true due to coercion, but if ([]) is also true because an empty array is truthy. This is why you should use === for comparisons.
  • Q6. What is the difference between null and undefined?
    undefined means a variable has been declared but not yet assigned a value — it's the JavaScript engine's default. null is an intentional assignment meaning "no value" — it's a deliberate choice by the programmer. typeof undefined is "undefined"; typeof null is "object" (a famous JS bug that can never be fixed for backward compatibility). null == undefined is true (loose equality), but null === undefined is false. Use null when you intentionally want to represent absence; undefined appears naturally when values aren't set.
  • Q7. What is event bubbling and how does event delegation use it?
    Event bubbling means that when an event occurs on an element, it first runs that element's handlers, then bubbles up to the parent's handlers, then grandparent's, all the way to the document. Event delegation exploits this by attaching a single listener to a parent element instead of adding listeners to each child. When a child is clicked, the event bubbles to the parent listener, which checks event.target to see which child triggered it. Benefits: uses less memory, automatically handles dynamically added children, cleaner code. e.stopPropagation() can prevent bubbling when needed.
  • Q8. What is the difference between textContent and innerHTML?
    textContent gets or sets the plain text content of an element — any HTML tags are treated as literal text characters, not parsed. It's safe from XSS. innerHTML gets or sets the HTML markup — tags inside are parsed and rendered. Setting innerHTML with user-provided content is dangerous (XSS attack vector — malicious scripts can be injected). Use textContent for text; use innerHTML only with trusted, sanitized content. For inserting new elements safely, use createElement + appendChild instead of innerHTML.
