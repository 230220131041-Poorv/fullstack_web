State vs Props
The two ways data exists in a React component — know the difference cold

The Core Distinction

Props are data passed INTO a component from its parent — read-only, controlled from outside. State is data OWNED and managed BY the component itself — it can change over time (user clicks, typing, API responses) and when it changes, React automatically re-renders that component to reflect the new value.

Mental model: Props are like function parameters — passed in, can't be changed by the function itself. State is like a function's own local variable that, when updated, triggers React to re-run the component and produce updated UI.
Props

✦ Passed from parent
✦ Read-only (immutable)
✦ Component cannot change its own props
✦ Changes when PARENT re-renders with new values

State

✦ Owned by the component itself
✦ Mutable (via setter function)
✦ Component fully controls its own state
✦ Changing it triggers a re-render of THAT component

The useState Hook
Giving components memory — the most-used hook in all of React

What is a Hook?

A Hook is a special function that lets you "hook into" React features (state, lifecycle, context) from function components. Hooks always start with use. useState is the hook that gives a component the ability to remember a value across re-renders.

The Two Rules of Hooks (asked constantly in interviews):
1. Only call hooks at the TOP LEVEL of a component — never inside loops, conditions, or nested functions.
2. Only call hooks from React function components or custom hooks — never from regular JS functions.
JSX — useState Basics
import { useState } from 'react';

function Counter() {
  // useState(initialValue) returns [currentValue, setterFunction]
  const [count, setCount] = useState(0);  // 0 = initial value, ONLY used on first render

  function increment() {
    setCount(count + 1);  // updates state AND triggers a re-render
  }

  return (
    <div>
      <p>Count: {count}</p>
      <button onClick={increment}>+1</button>
    </div>
  );
}

// ── ARRAY DESTRUCTURING NAMING CONVENTION ──
// [value, setValue] — always pair "noun" with "setNoun"
const [name, setName]       = useState('');
const [isOpen, setIsOpen]   = useState(false);
const [items, setItems]     = useState([]);
const [user, setUser]       = useState(null);

// ── INLINE event handler (no separate function needed) ──
function Counter() {
  const [count, setCount] = useState(0);
  return (
    <button onClick={() => setCount(count + 1)}>
      Clicked {count} times
    </button>
  );
}

// ── FUNCTIONAL UPDATES — when new state depends on OLD state ──
function increment() {
  setCount(count + 1);  // works, but "count" might be stale in some scenarios
}
function incrementSafe() {
  setCount(prevCount => prevCount + 1);  // ✅ ALWAYS correct — gets the LATEST state
}

// Why this matters: calling setCount 3 times in a row
function incrementThreeTimes() {
  setCount(count + 1); // these all read the SAME "count" from this render
  setCount(count + 1); // → count only increases by 1 total, not 3! (common bug)
  setCount(count + 1);
}
function incrementThreeTimesCorrect() {
  setCount(c => c + 1); // ✅ each call uses the latest value
  setCount(c => c + 1); // → count correctly increases by 3
  setCount(c => c + 1);
}

State Updates Are Asynchronous (Batched)

Calling the setter function doesn't immediately change the variable in the current render — it schedules a re-render. React batches multiple state updates together for performance, applying them before the next render.

JSX — State Updates Don't Happen Immediately
function handleClick() {
  setCount(count + 1);
  console.log(count);  // ❌ still shows the OLD value! State hasn't updated yet
  // React schedules the update, re-renders AFTER this function finishes
}

// If you need to react to the new value, use useEffect (covered below)
// or compute it manually with the same logic used in the setter

State with Objects & Arrays
Immutability is mandatory — never mutate state directly

Why You Must Never Mutate State Directly

React detects changes by comparing references (using Object.is essentially), not by deeply inspecting values. If you mutate an object/array in place, the reference stays the SAME, so React doesn't know anything changed and won't re-render. You must always create a NEW object/array (using spread syntax) when updating state.

JSX — Updating Object State (Immutably)
function ProfileForm() {
  const [user, setUser] = useState({ name: 'Poorv', age: 21, city: 'Junagadh' });

  // ❌ WRONG — mutating directly, React won't detect this change
  function updateNameWrong(newName) {
    user.name = newName;     // mutates the SAME object — no re-render!
    setUser(user);            // same reference — React sees "no change"
  }

  // ✅ CORRECT — spread to create a NEW object
  function updateName(newName) {
    setUser({ ...user, name: newName });  // new object, only 'name' changes
  }

  function updateNested(newCity) {
    setUser(prevUser => ({ ...prevUser, city: newCity })); // functional update, safer
  }

  return (
    <input
      value={user.name}
      onChange={(e) => updateName(e.target.value)}
    />
  );
}
JSX — Updating Array State (Immutably)
function TodoApp() {
  const [todos, setTodos] = useState([
    { id: 1, text: 'Learn React', done: false },
    { id: 2, text: 'Build a project', done: false }
  ]);

  // ── ADDING an item ──
  function addTodo(text) {
    const newTodo = { id: Date.now(), text, done: false };
    setTodos([...todos, newTodo]);     // ✅ new array, old items + 1 new
    // ❌ NEVER: todos.push(newTodo) — mutates the original array
  }

  // ── REMOVING an item ──
  function removeTodo(id) {
    setTodos(todos.filter(todo => todo.id !== id));  // ✅ new array without that item
  }

  // ── UPDATING an item ──
  function toggleTodo(id) {
    setTodos(todos.map(todo =>
      todo.id === id
        ? { ...todo, done: !todo.done }  // new object for the matching item
        : todo                          // unchanged items stay the same reference
    ));
  }

  return (
    <ul>
      {todos.map(todo => (
        <li key={todo.id} onClick={() => toggleTodo(todo.id)}>
          {todo.done ? '✅' : '⬜'} {todo.text}
          <button onClick={(e) => { e.stopPropagation(); removeTodo(todo.id); }}>✕</button>
        </li>
      ))}
    </ul>
  );
}
The #1 React bug for beginners: Mutating state directly (push, pop, splice, sort on arrays; direct property assignment on objects). It often "seems to work" in dev because of re-renders triggered by OTHER state changes nearby, masking the bug — then fails mysteriously later. ALWAYS use spread syntax, map, filter, or concat to produce new arrays/objects.

The useEffect Hook
Running code in response to renders — for side effects like API calls

What is a Side Effect?

A side effect is anything that reaches "outside" the pure render process — fetching data, subscribing to events, manually changing the DOM, setting timers, logging to analytics. useEffect lets you run this code AFTER React has rendered/updated the DOM, separate from the rendering logic itself.

JSX — useEffect Basics
import { useState, useEffect } from 'react';

function Example() {
  const [count, setCount] = useState(0);

  // useEffect(callback, dependencyArray)
  useEffect(() => {
    console.log('Effect ran! Count is:', count);
    document.title = `Count: ${count}`;  // example side effect: update browser tab title
  });  // NO dependency array → runs after EVERY render

  return <button onClick={() => setCount(count + 1)}>Count: {count}</button>;
}

The Dependency Array — Controls WHEN the Effect Runs

The second argument to useEffect is critical. It tells React when to re-run the effect.

Dependency ArrayWhen it runs
No array at allAfter EVERY single render — rarely what you want
[] (empty array)Only ONCE, after the initial render — perfect for "on mount" logic like initial data fetch
[count]After initial render, AND whenever 'count' changes between renders
[a, b]Re-runs if EITHER 'a' OR 'b' changes
JSX — The 3 Dependency Array Patterns
// ── PATTERN 1: Run on every render (rarely needed) ──
useEffect(() => {
  console.log('Runs after every render');
});

// ── PATTERN 2: Run ONCE on mount (most common for data fetching) ──
function UserProfile({ userId }) {
  const [user, setUser] = useState(null);

  useEffect(() => {
    fetch(`/api/users/${userId}`)
      .then(res => res.json())
      .then(data => setUser(data));
  }, []); // ⚠️ empty array — runs ONCE when component first mounts

  return <div>{user?.name}</div>;
}

// ── PATTERN 3: Re-run when a specific value changes ──
function UserProfile({ userId }) {
  const [user, setUser] = useState(null);

  useEffect(() => {
    fetch(`/api/users/${userId}`)
      .then(res => res.json())
      .then(data => setUser(data));
  }, [userId]); // ✅ re-fetches whenever userId prop changes (e.g. navigating to a different user)

  return <div>{user?.name}</div>;
}

// ── ⚠️ COMMON BUG: missing dependency causes stale data ──
useEffect(() => {
  fetch(`/api/users/${userId}`).then(...);
}, []); // ❌ userId is used inside but NOT in dependency array
       // → effect never re-runs even if userId changes! ESLint will warn about this.

Cleanup Functions
Preventing memory leaks — the return function inside useEffect

Why Cleanup Matters

Some effects create ongoing subscriptions, timers, or event listeners that must be removed when the component unmounts (or before the effect re-runs) — otherwise you get memory leaks, duplicate listeners, or "setState on unmounted component" warnings. You handle this by returning a cleanup function from your effect.

JSX — Cleanup Functions
// ── EXAMPLE 1: Timer that must be cleared ──
function Clock() {
  const [time, setTime] = useState(new Date());

  useEffect(() => {
    const intervalId = setInterval(() => {
      setTime(new Date());
    }, 1000);

    // CLEANUP FUNCTION — runs when component unmounts OR before effect re-runs
    return () => {
      clearInterval(intervalId);  // ✅ stops the timer — prevents memory leak
    };
  }, []); // runs once; cleanup runs when component is removed from the page

  return <p>{time.toLocaleTimeString()}</p>;
}

// ── EXAMPLE 2: Event listener cleanup ──
function WindowSizeTracker() {
  const [width, setWidth] = useState(window.innerWidth);

  useEffect(() => {
    function handleResize() {
      setWidth(window.innerWidth);
    }

    window.addEventListener('resize', handleResize);

    return () => {
      window.removeEventListener('resize', handleResize); // ✅ remove listener on unmount
    };
  }, []);

  return <p>Window width: {width}px</p>;
}

// ── EXAMPLE 3: Preventing "setState on unmounted component" with fetch ──
function UserProfile({ userId }) {
  const [user, setUser] = useState(null);

  useEffect(() => {
    let isCancelled = false;

    fetch(`/api/users/${userId}`)
      .then(res => res.json())
      .then(data => {
        if (!isCancelled) setUser(data); // only update if still mounted/relevant
      });

    return () => { isCancelled = true; }; // mark as cancelled on cleanup
  }, [userId]);

  return <div>{user?.name}</div>;
}
Cleanup timing rule: The cleanup function runs (1) right before the effect runs again due to a dependency change, and (2) when the component unmounts entirely. This pattern of "cleanup-then-rerun" is why intervals/listeners never duplicate even as dependencies change repeatedly.

Lifting State Up
Sharing state between sibling components

The Problem It Solves

When two sibling components need to share or stay in sync with the same piece of state, the solution is to move ("lift") that state UP to their closest common parent, then pass it down as props — along with a function to update it.

JSX — Lifting State Up Example
// ── PROBLEM: SearchBar and ResultsList are siblings, but ResultsList
//    needs to know what SearchBar's input value is ──

// ── SOLUTION: Lift the 'query' state up to the shared parent ──
function SearchPage() {
  const [query, setQuery] = useState('');  // state lives in the PARENT

  return (
    <div>
      <SearchBar query={query} onQueryChange={setQuery} />
      <ResultsList query={query} />
    </div>
  );
}

function SearchBar({ query, onQueryChange }) {
  return (
    <input
      value={query}
      onChange={(e) => onQueryChange(e.target.value)}
      placeholder="Search products..."
    />
  );
}

function ResultsList({ query }) {
  const filtered = allProducts.filter(p =>
    p.name.toLowerCase().includes(query.toLowerCase())
  );
  return (
    <ul>
      {filtered.map(p => <li key={p.id}>{p.name}</li>)}
    </ul>
  );
}
// Now both SearchBar and ResultsList stay in sync through their shared parent's state
🏢 Real World

This pattern is everywhere

Lifting state up is exactly how filters + product lists, tabs + tab content, and form steps + progress indicators stay synchronized in real applications. Once components need to share state across more distant parts of the tree, teams reach for Context API (Day 10) or external state libraries.

Controlled Forms
Forms where React state IS the single source of truth

Controlled vs Uncontrolled Inputs

In a controlled component, the input's value is driven entirely by React state — the DOM input's value always equals the state, and every keystroke updates state via onChange. This is the standard, recommended React pattern (vs "uncontrolled" inputs that manage their own internal DOM state, accessed via refs).

JSX — Complete Controlled Form
function SignupForm() {
  const [formData, setFormData] = useState({
    name: '',
    email: '',
    role: 'user',
    subscribe: false
  });
  const [errors, setErrors] = useState({});

  // ── ONE handler for ALL inputs (using the 'name' attribute) ──
  function handleChange(e) {
    const { name, value, type, checked } = e.target;
    setFormData(prev => ({
      ...prev,
      [name]: type === 'checkbox' ? checked : value  // computed key!
    }));
  }

  function handleSubmit(e) {
    e.preventDefault();  // stop page reload

    const newErrors = {};
    if (!formData.name.trim()) newErrors.name = 'Name is required';
    if (!formData.email.includes('@')) newErrors.email = 'Invalid email';
    setErrors(newErrors);

    if (Object.keys(newErrors).length === 0) {
      console.log('Submitting:', formData);
      // send to API here
    }
  }

  return (
    <form onSubmit={handleSubmit}>
      <input
        name="name"
        value={formData.name}      // ← controlled: value comes FROM state
        onChange={handleChange}  // ← every keystroke updates state
        placeholder="Name"
      />
      {errors.name && <span className="error">{errors.name}</span>}

      <input
        name="email"
        type="email"
        value={formData.email}
        onChange={handleChange}
        placeholder="Email"
      />
      {errors.email && <span className="error">{errors.email}</span>}

      <select name="role" value={formData.role} onChange={handleChange}>
        <option value="user">User</option>
        <option value="admin">Admin</option>
      </select>

      <label>
        <input
          type="checkbox"
          name="subscribe"
          checked={formData.subscribe}   // checkboxes use 'checked' not 'value'
          onChange={handleChange}
        />
        Subscribe to newsletter
      </label>

      <button type="submit">Sign Up</button>
    </form>
  );
}

useRef & A Glimpse of Other Hooks
Accessing DOM nodes directly and storing values that don't trigger re-renders

useRef — A Mutable Box That Doesn't Cause Re-renders

useRef creates a mutable object that persists across renders, but — unlike state — changing it does NOT trigger a re-render. It's most commonly used to directly access a DOM element (like focusing an input) or to store a value you need to remember without it affecting rendering.

JSX — useRef Examples
import { useRef, useEffect } from 'react';

// ── EXAMPLE 1: Focusing an input on mount ──
function SearchInput() {
  const inputRef = useRef(null);  // starts as null

  useEffect(() => {
    inputRef.current.focus();  // .current holds the actual DOM node after mount
  }, []);

  return <input ref={inputRef} placeholder="Auto-focused!" />;
}

// ── EXAMPLE 2: Storing a value WITHOUT causing re-renders ──
function RenderCounter() {
  const renderCount = useRef(0);
  renderCount.current += 1;  // changing this does NOT trigger a re-render

  return <p>This component has rendered {renderCount.current} times</p>;
}

// ── EXAMPLE 3: Storing the previous value of a state variable ──
function Counter() {
  const [count, setCount] = useState(0);
  const prevCountRef = useRef();

  useEffect(() => {
    prevCountRef.current = count; // save AFTER render, becomes "previous" on next render
  }, [count]);

  return <p>Now: {count}, Before: {prevCountRef.current}</p>;
}
useState vs useRef

useState: changing it triggers a re-render; value is "visible" in JSX.
useRef: changing .current does NOT trigger a re-render; used for DOM access or "remembering" values silently.

Real Project: Complete Todo App
Combining useState, useEffect, controlled forms, and list rendering
🏢 Real World

A Todo app is the "Hello World" of React interviews

It's commonly assigned in technical screening rounds because it tests state management, list rendering, controlled inputs, and localStorage persistence — all in one small project.

JSX — Complete Todo App with localStorage Persistence
import { useState, useEffect } from 'react';

function TodoApp() {
  const [todos, setTodos] = useState(() => {
    // Lazy initial state — function runs ONLY on first render (good for expensive ops)
    const saved = localStorage.getItem('todos');
    return saved ? JSON.parse(saved) : [];
  });
  const [input, setInput] = useState('');
  const [filter, setFilter] = useState('all'); // 'all' | 'active' | 'completed'

  // ── Persist todos to localStorage whenever they change ──
  useEffect(() => {
    localStorage.setItem('todos', JSON.stringify(todos));
  }, [todos]);

  function addTodo(e) {
    e.preventDefault();
    if (!input.trim()) return;

    setTodos(prev => [...prev, { id: Date.now(), text: input.trim(), done: false }]);
    setInput('');  // clear the input after adding
  }

  function toggleTodo(id) {
    setTodos(prev => prev.map(t =>
      t.id === id ? { ...t, done: !t.done } : t
    ));
  }

  function deleteTodo(id) {
    setTodos(prev => prev.filter(t => t.id !== id));
  }

  const filteredTodos = todos.filter(t => {
    if (filter === 'active') return !t.done;
    if (filter === 'completed') return t.done;
    return true; // 'all'
  });

  const activeCount = todos.filter(t => !t.done).length;

  return (
    <div className="todo-app">
      <h1>My Todos</h1>

      <form onSubmit={addTodo}>
        <input
          value={input}
          onChange={(e) => setInput(e.target.value)}
          placeholder="What needs to be done?"
        />
        <button type="submit">Add</button>
      </form>

      <div className="filters">
        {['all', 'active', 'completed'].map(f => (
          <button
            key={f}
            className={filter === f ? 'active' : ''}
            onClick={() => setFilter(f)}
          >
            {f}
          </button>
        ))}
      </div>

      {filteredTodos.length === 0 ? (
        <p className="empty">No todos here!</p>
      ) : (
        <ul>
          {filteredTodos.map(todo => (
            <li key={todo.id} className={todo.done ? 'done' : ''}>
              <input
                type="checkbox"
                checked={todo.done}
                onChange={() => toggleTodo(todo.id)}
              />
              <span>{todo.text}</span>
              <button onClick={() => deleteTodo(todo.id)}>✕</button>
            </li>
          ))}
        </ul>
      )}

      <p className="counter">{activeCount} tasks left</p>
    </div>
  );
}

export default TodoApp;

Placement Interview Q&A
Top React state & hooks questions for company interviews
  • Q1. What is the difference between state and props?
    Props are passed into a component from its parent and are read-only — a component cannot modify its own props. State is data that a component owns and manages internally, and it CAN be changed via its setter function — when it changes, React re-renders that component. Props flow down (parent to child, unidirectional); state is local to wherever it's declared. If a child needs to "change" something, the parent passes down a function as a prop that the child calls — the actual state mutation happens wherever the state was declared with useState.
  • Q2. Why can't you mutate state directly in React? Give an example.
    React determines whether to re-render by comparing the reference of the new state to the old state (not deeply inspecting values). If you mutate an array or object in place (e.g., todos.push(newTodo) or user.name = 'New'), the reference stays exactly the same, so React doesn't detect any change and won't re-render — even though the data technically changed. You must always create a NEW object/array using spread syntax, map, or filter, e.g., setTodos([...todos, newTodo]) or setUser({...user, name: 'New'}), so React sees a new reference and knows to update the UI.
  • Q3. What does the dependency array in useEffect control?
    It controls when the effect re-runs. No array at all means the effect runs after every render. An empty array [] means it runs only once, after the initial render (commonly used for one-time data fetching). An array with values like [userId] means the effect re-runs whenever any value in that array changes between renders, in addition to running once on mount. Forgetting to include a value that the effect actually uses is a common bug — it causes the effect to use a stale, outdated value because it never re-runs to pick up the new one.
  • Q4. What is a cleanup function in useEffect and when do you need one?
    A cleanup function is returned from inside the useEffect callback, and React runs it automatically right before the effect re-runs (due to a dependency change) and when the component unmounts. You need one whenever your effect sets up something ongoing that should be torn down — like a setInterval/setTimeout timer, an event listener (addEventListener), or a subscription. Without cleanup, these continue running even after the component is gone, causing memory leaks, duplicate listeners, or "setState called on an unmounted component" warnings.
  • Q5. What is "lifting state up" and why would you do it?
    Lifting state up means moving a piece of state from a child component to its closest common parent, so that multiple sibling components can share and stay synchronized with that same state. For example, if a SearchBar and a ResultsList are siblings but ResultsList needs to know what's typed in SearchBar, you move the "query" state to their shared parent component, then pass the value down to ResultsList as a prop and pass an update function down to SearchBar as a prop. This keeps a single source of truth instead of each component having its own disconnected copy of the data.
  • Q6. What does it mean for an input to be "controlled" in React?
    A controlled input is one whose value is fully driven by React state — the input's "value" attribute is set to a state variable, and every change (onChange) updates that state variable, which then re-renders the input with the new value. This makes React state the single source of truth for the form data at all times, which makes validation, conditional logic, and submitting the data straightforward. The opposite is an "uncontrolled" input, which manages its own value internally in the DOM and is accessed via a ref only when needed (e.g., on submit), rather than on every keystroke.
  • Q7. What is the difference between useState and useRef?
    useState stores a value AND triggers a re-render of the component whenever that value is updated via its setter — the value is meant to be reflected visually in the JSX. useRef also persists a value across renders, but updating ref.current does NOT trigger a re-render — it's used for accessing DOM nodes directly (like focusing an input) or remembering a value (like a previous state, a timer ID, or a render count) without needing the component to visually update because of it.
  • Q8. Why does calling a state setter not immediately update the variable in the same function?
    State updates in React are asynchronous and batched for performance — calling setCount(count + 1) schedules a re-render rather than mutating the variable synchronously in place. So console.log(count) immediately after calling the setter still shows the OLD value, because the actual update and re-render happen after the current function finishes executing. If you need to perform multiple sequential updates based on the latest value (especially when calling the setter multiple times in a row), use the functional form: setCount(prevCount => prevCount + 1), which always receives the most up-to-date state value, avoiding bugs where rapid updates seem to "skip" increments.
