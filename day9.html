What is React?
A JavaScript library for building user interfaces — not a framework

React's Core Idea

React is a JavaScript library (not a full framework like Angular) created by Facebook/Meta in 2013, for building user interfaces out of reusable components. Its core philosophy: describe WHAT the UI should look like for any given state, and React figures out HOW to efficiently update the actual DOM to match.

Before React, developers manually manipulated the DOM with code like document.getElementById(...).innerHTML = ... everywhere — which becomes unmanageable as apps grow. React introduced a declarative approach: you write components that describe the UI based on data, and React handles all the DOM updates automatically when that data changes.

Declarative vs Imperative: Imperative = "click button → find the element → manually change its text → manually update the counter display." Declarative (React) = "the counter display should ALWAYS show {count}" — you just update count, and React re-renders the UI to match automatically.

The Virtual DOM

React maintains a lightweight, in-memory copy of the real DOM called the Virtual DOM. When state changes, React: (1) creates a new Virtual DOM tree representing the updated UI, (2) compares ("diffs") it against the previous Virtual DOM tree, (3) calculates the minimal set of actual DOM changes needed, and (4) applies only those changes to the real DOM. This is far faster than re-rendering the entire page on every change.

Why Companies Want React Developers

React (and its ecosystem — Next.js, React Native) dominates the frontend job market. Component-based architecture means teams can build, test, and reuse UI pieces independently. Most Ahmedabad web development job postings for "Full Stack Developer" or "Frontend Developer" explicitly require React experience.

Setting Up a React Project
Vite — the modern, fast way to start a React app

Vite vs Create React App

Vite is the current industry standard for starting new React projects — it's dramatically faster than the older create-react-app (which is now deprecated) because it uses native ES modules during development instead of bundling everything upfront.

Bash — Creating a New React App
# Create a new React project with Vite
npm create vite@latest my-app -- --template react

cd my-app
npm install
npm run dev          # starts dev server, usually at http://localhost:5173

# Production build
npm run build        # creates optimized dist/ folder for deployment

# ── PROJECT STRUCTURE ──
/*
my-app/
├── node_modules/
├── public/              ← static assets served as-is
├── src/
│   ├── App.jsx           ← root component
│   ├── main.jsx          ← entry point, renders App into the DOM
│   ├── index.css
│   └── components/       ← your reusable components live here
├── index.html            ← single HTML file (SPA)
├── package.json
└── vite.config.js
*/
JSX — Entry Point (main.jsx)
import { StrictMode } from 'react';
import { createRoot } from 'react-dom/client';
import App from './App.jsx';

// Find the <div id="root"> in index.html and mount the React app inside it
createRoot(document.getElementById('root')).render(
  <StrictMode>
    <App />
  </StrictMode>
);
// StrictMode highlights potential problems during development (no effect in production)

JSX — Deep Dive
JavaScript + XML — writing UI structure directly in JavaScript

What is JSX?

JSX (JavaScript XML) is a syntax extension that lets you write HTML-like markup directly inside JavaScript. Browsers can't run JSX directly — a build tool (Vite, using Babel/esbuild under the hood) compiles JSX into plain JavaScript function calls before the code runs.

Under the hood: <h1>Hello</h1> compiles to React.createElement('h1', null, 'Hello'). JSX is just syntax sugar over regular function calls that create React elements (plain JS objects describing what should appear on screen).
JSX — Rules and Syntax
// ── RULE 1: Must return a SINGLE root element ──
function Bad() {
  return (
    <h1>Title</h1>
    <p>Text</p>  // ❌ Error — two root elements, JSX needs ONE parent
  );
}

function Good() {
  return (
    <div>
      <h1>Title</h1>
      <p>Text</p>
    </div>  // ✅ wrapped in one parent div
  );
}

// Fragment (<>...</>) — group elements WITHOUT adding an extra DOM node
function Better() {
  return (
    <>
      <h1>Title</h1>
      <p>Text</p>
    </>  // ✅ no extra <div> wrapper added to the actual DOM
  );
}

// ── RULE 2: Use {} to embed JavaScript expressions ──
function Greeting() {
  const name = 'Poorv';
  const age  = 21;
  return (
    <div>
      <h1>Hello, {name}!</h1>          // embeds a variable
      <p>Next year you'll be {age + 1}</p>  // embeds an EXPRESSION
      <p>{name.toUpperCase()}</p>           // can call functions/methods
    </div>
  );
}
// ⚠️ JSX can only embed EXPRESSIONS, not statements
// {if (x) {...}} ❌ doesn't work — use ternary instead: {x ? ... : ...}

// ── RULE 3: className NOT class, htmlFor NOT for (JS reserved words) ──
function Styled() {
  return (
    <div className="card">          // NOT class= (class is a JS reserved word)
      <label htmlFor="email">Email</label>  // NOT for=
      <input id="email" onChange={handleChange} />  // camelCase events
    </div>
  );
}

// ── RULE 4: Self-closing tags need a slash ──
<img src="photo.jpg" />   // ✅ correct
// <img src="photo.jpg">     ❌ missing slash — JSX requires it for void elements

// ── RULE 5: Inline styles use an OBJECT, not a string ──
function InlineStyled() {
  const styles = { color: 'blue', fontSize: '20px' }; // camelCase keys!
  return <p style={styles}>Styled text</p>;
  // or inline directly: <p style={{ color: 'blue', fontSize: '20px' }}>
  // (double braces: outer = JS expression, inner = the object itself)
}

// ── RULE 6: Comments inside JSX use {/* */} ──
function Commented() {
  return (
    <div>
      {/* This is a JSX comment — NOT // or <!-- --> */}
      <p>Content</p>
    </div>
  );
}

Components
The fundamental building block of every React application

What is a Component?

A component is a reusable, independent piece of UI — essentially a JavaScript function that returns JSX describing what should render. Component names MUST start with a capital letter (this is how React/JSX distinguishes a custom component <UserCard /> from a regular HTML tag <div>).

JSX — Function Components
// ── BASIC FUNCTION COMPONENT (the modern standard — always use this) ──
function Welcome() {
  return <h1>Welcome to my site!</h1>;
}

// Arrow function syntax (equally valid, very common)
const Welcome2 = () => {
  return <h1>Welcome!</h1>;
};

// Implicit return (no curly braces — for simple components)
const Welcome3 = () => <h1>Welcome!</h1>;

// ── USING a component — just like an HTML tag ──
function App() {
  return (
    <div>
      <Welcome />    // renders the Welcome component
      <Welcome />    // reusable — render it as many times as needed
      <Welcome />
    </div>
  );
}

// ── EXPORTING & IMPORTING components across files (one component per file) ──
// In Welcome.jsx:
function Welcome() {
  return <h1>Welcome!</h1>;
}
export default Welcome;     // default export — one per file

// In App.jsx:
import Welcome from './Welcome.jsx';  // import it wherever needed

// ── COMPONENT FILE NAMING CONVENTION ──
// PascalCase for both component name AND file name:
// UserProfile.jsx, ProductCard.jsx, NavBar.jsx

Component Tree — How Apps Are Structured

A React app is a tree of components, starting from one root (usually App) and branching into smaller, focused components — similar to how HTML elements nest, but each piece is independent and reusable.

Props
Passing data from parent to child components

What are Props?

Props (properties) are how data flows from a parent component to a child component — similar to function arguments. Props are read-only — a component must never modify its own props directly (this is one of React's core rules). Data flows in ONE direction: parent → child (called "unidirectional data flow").

JSX — Props Complete
// ── PASSING props (parent component) ──
function App() {
  return (
    <div>
      <UserCard name="Poorv" age={21} isStudent={true} />
      <UserCard name="Riya" age={23} isStudent={false} />
    </div>
  );
}

// ── RECEIVING props (child component) — via the function parameter ──
function UserCard(props) {
  return (
    <div className="card">
      <h3>{props.name}</h3>
      <p>Age: {props.age}</p>
      <p>{props.isStudent ? 'Student' : 'Professional'}</p>
    </div>
  );
}

// ── DESTRUCTURING props (the modern, preferred style — used constantly) ──
function UserCard({ name, age, isStudent }) {  // destructure right in the parameter
  return (
    <div className="card">
      <h3>{name}</h3>
      <p>Age: {age}</p>
      <p>{isStudent ? 'Student' : 'Professional'}</p>
    </div>
  );
}

// ── DEFAULT PROP VALUES ──
function Avatar({ src, size = 50, alt = 'User avatar' }) {
  return <img src={src} width={size} alt={alt} />;
}
<Avatar src="photo.jpg" />            // uses default size=50
<Avatar src="photo.jpg" size={100} />  // overrides default

// ── PASSING an OBJECT as a single prop ──
const user = { name: 'Poorv', age: 21, city: 'Junagadh' };
<UserCard user={user} />             // pass whole object as ONE prop

function UserCard({ user }) {            // receive the object
  return <h3>{user.name} from {user.city}</h3>;
}

// Or spread individual values as separate props:
<UserCard {...user} />           // equivalent to name={user.name} age={user.age} city={user.city}

// ── PASSING FUNCTIONS as props (for child-to-parent communication) ──
function App() {
  function handleLike(productId) {
    console.log(`Liked product ${productId}`);
  }
  return <ProductCard id={1} onLike={handleLike} />;
}

function ProductCard({ id, onLike }) {
  return <button onClick={() => onLike(id)}>Like</button>;
}
// This is THE pattern for letting a child trigger something in its parent

// ── children prop — special prop for nested JSX content ──
function Card({ children }) {
  return <div className="card">{children}</div>;
}
<Card>
  <h2>Title</h2>
  <p>This whole block becomes props.children</p>
</Card>
Critical rule: Props are READ-ONLY. A component must NEVER modify its own props (e.g., props.name = 'New' ❌). If a child needs to change something, the parent passes down a FUNCTION as a prop, and the child calls that function — the actual state change happens in the parent (or wherever the state lives). This is the foundation for understanding state management, covered tomorrow.

Component Composition
Building complex UIs from small, focused pieces

Composition Over Configuration

Instead of building one giant component with dozens of props to control every variation, React encourages composition — building small, focused components and combining them. This is more flexible and maintainable.

JSX — Composition Example
// ── BREAKING a page into small, reusable components ──

function Avatar({ src, alt }) {
  return <img className="avatar" src={src} alt={alt} />;
}

function Badge({ text, color }) {
  return <span className={`badge badge-${color}`}>{text}</span>;
}

function UserHeader({ user }) {
  return (
    <div className="user-header">
      <Avatar src={user.avatar} alt={user.name} />
      <h2>{user.name}</h2>
      <Badge text="Pro Member" color="gold" />
    </div>
  );
}

// ── COMPOSING with children (slot pattern — very flexible) ──
function Modal({ title, children, onClose }) {
  return (
    <div className="modal-overlay">
      <div className="modal">
        <div className="modal-header">
          <h2>{title}</h2>
          <button onClick={onClose}>✕</button>
        </div>
        <div className="modal-body">
          {children}  // ← whatever content the caller passes in goes HERE
        </div>
      </div>
    </div>
  );
}

// ── REUSING the same Modal shell with DIFFERENT content ──
<Modal title="Confirm Delete" onClose={closeModal}>
  <p>Are you sure you want to delete this item?</p>
  <button>Yes, delete</button>
</Modal>

<Modal title="Edit Profile" onClose={closeModal}>
  <ProfileForm />
</Modal>
// Same Modal shell, completely different content — that's the power of composition

Rendering Lists & Keys
Turning arrays of data into UI — used in literally every app

map() to Render Lists

To render a list of items in React, you use the .map() array method (covered Day 5) to transform an array of data into an array of JSX elements.

JSX — Rendering Lists with Keys
const products = [
  { id: 1, name: 'Laptop', price: 55000 },
  { id: 2, name: 'Mouse',  price: 499 },
  { id: 3, name: 'Keyboard', price: 1299 },
];

function ProductList() {
  return (
    <ul>
      {products.map((product) => (
        <li key={product.id}>           // 'key' is REQUIRED — unique per item
          {product.name} - ₹{product.price}
        </li>
      ))}
    </ul>
  );
}

// ── USING a separate component PER item (more realistic pattern) ──
function ProductCard({ product }) {
  return (
    <div className="product-card">
      <h3>{product.name}</h3>
      <p>₹{product.price}</p>
    </div>
  );
}

function ProductGrid() {
  return (
    <div className="grid">
      {products.map((product) => (
        <ProductCard key={product.id} product={product} />
      ))}
    </div>
  );
}

Why Keys Matter So Much

The key prop helps React identify which items have changed, been added, or removed between re-renders — without it, React falls back to re-rendering the entire list inefficiently, and can cause real bugs with component state mismatching items.

⚠️ Never use array index as a key if the list can be reordered, filtered, or items can be added/removed from the middle — key={index} causes React to mismatch state across re-renders (e.g., text typed into the wrong input after deleting an item above it). ALWAYS use a stable, unique ID from your data (like a database id) as the key. Index-as-key is only acceptable for completely static lists that never change order or length.

Conditional Rendering
Showing different UI based on data — every app needs this
JSX — All Conditional Rendering Patterns
// ── PATTERN 1: && for "render only if true" ──
function Notification({ hasUnread, count }) {
  return (
    <div>
      {hasUnread && <span className="badge">{count}</span>}
    </div>
  );
}
// ⚠️ GOTCHA: if count is 0, {0 && <Badge />} renders the LITERAL "0" on screen!
// Because 0 is falsy but it's still a renderable value. Fix: {count > 0 && <Badge />}

// ── PATTERN 2: Ternary for "render A or B" ──
function LoginButton({ isLoggedIn }) {
  return (
    <div>
      {isLoggedIn ? (
        <button>Log Out</button>
      ) : (
        <button>Log In</button>
      )}
    </div>
  );
}

// ── PATTERN 3: Early return for "don't render anything" ──
function UserGreeting({ user }) {
  if (!user) {
    return null;  // renders nothing — completely valid in React
  }
  return <h1>Welcome back, {user.name}!</h1>;
}

// ── PATTERN 4: if/else BEFORE the return (for complex logic) ──
function StatusMessage({ status }) {
  let message;
  if (status === 'loading') {
    message = <p>Loading...</p>;
  } else if (status === 'error') {
    message = <p className="error">Something went wrong</p>;
  } else {
    message = <p>All good!</p>;
  }
  return <div>{message}</div>;
}

// ── PATTERN 5: switch for multiple distinct states ──
function OrderStatus({ status }) {
  switch (status) {
    case 'pending':   return <span className="badge-yellow">Pending</span>;
    case 'shipped':   return <span className="badge-blue">Shipped</span>;
    case 'delivered': return <span className="badge-green">Delivered</span>;
    default:         return <span>Unknown</span>;
  }
}

Real Component: Product Catalog
Combining JSX, components, props, lists, and conditionals into a complete feature
🏢 Real World

This is the exact structure companies expect you to build

A product catalog page — search, filter, list rendering, conditional empty states — is one of the most common assignments given in technical interviews and real e-commerce projects at Ahmedabad companies.

JSX — Complete Product Catalog (no state yet — that's Day 9)
// ── ProductCard.jsx ──
function ProductCard({ product, onAddToCart }) {
  const { name, price, image, inStock } = product;

  return (
    <div className="product-card">
      <img src={image} alt={name} />
      <h3>{name}</h3>
      <p className="price">₹{price.toLocaleString()}</p>

      {inStock ? (
        <button onClick={() => onAddToCart(product)}>
          Add to Cart
        </button>
      ) : (
        <button disabled>Out of Stock</button>
      )}
    </div>
  );
}

export default ProductCard;

// ── ProductGrid.jsx ──
import ProductCard from './ProductCard';

function ProductGrid({ products, onAddToCart }) {
  if (products.length === 0) {
    return <p className="empty-state">No products found.</p>;
  }

  return (
    <div className="product-grid">
      {products.map((product) => (
        <ProductCard
          key={product.id}
          product={product}
          onAddToCart={onAddToCart}
        />
      ))}
    </div>
  );
}

export default ProductGrid;

// ── App.jsx ──
import ProductGrid from './ProductGrid';

const sampleProducts = [
  { id: 1, name: 'Wireless Mouse', price: 799, image: '/mouse.jpg', inStock: true },
  { id: 2, name: 'Mechanical Keyboard', price: 3499, image: '/keyboard.jpg', inStock: false },
];

function App() {
  function handleAddToCart(product) {
    console.log('Added to cart:', product.name);
  }

  return (
    <div className="app">
      <h1>Our Products</h1>
      <ProductGrid products={sampleProducts} onAddToCart={handleAddToCart} />
    </div>
  );
}

export default App;

Placement Interview Q&A
Top React fundamentals questions for Ahmedabad company interviews
  • Q1. What is JSX and how is it different from HTML?
    JSX (JavaScript XML) is a syntax extension that lets you write HTML-like markup inside JavaScript files. It's not actually HTML — it compiles to React.createElement() function calls via a build tool like Babel/Vite before the browser ever sees it. Key differences from HTML: attributes use camelCase (className not class, onClick not onclick), you embed JS expressions with curly braces {}, every element must be closed (including void elements like ), and a component must return a single root element (or a Fragment).
  • Q2. What is the difference between a React component and an HTML element in JSX?
    JSX distinguishes them by capitalization: lowercase tags like
    or

    are treated as regular HTML elements, while capitalized tags like are treated as references to your custom React components/functions. This is why component names and files MUST start with a capital letter (PascalCase) — if you name a component "usercard" instead of "UserCard", React/JSX will try to render it as an unknown HTML tag instead of your component.

  • Q3. What are props and what does it mean that they're "read-only"?
    Props (properties) are how a parent component passes data down to a child component, similar to function arguments. They are read-only/immutable from the child's perspective — a component must never directly modify its own props (e.g., props.value = 'new' is forbidden). This is part of React's unidirectional data flow principle: data flows down from parent to child, and if a child needs to "change" something, the parent passes a callback function as a prop, and the child calls that function to request the change — the actual mutation happens in the parent (or wherever the underlying state lives).
  • Q4. Why does React require a 'key' prop when rendering lists, and why shouldn't you use the array index as a key?
    The key prop helps React's reconciliation algorithm identify which list items have changed, been added, moved, or removed between renders, so it can efficiently update only what's necessary instead of re-rendering the entire list. Using the array index as a key is problematic when the list order can change (sorting, filtering, inserting/removing items from the middle) — because the index shifts but doesn't represent the same logical item anymore, React can mismatch internal state (like an input's typed value) to the wrong item after a reorder. Always use a stable, unique identifier from your data (like a database ID) as the key instead.
  • Q5. What is the children prop?
    children is a special, automatically-passed prop that contains whatever JSX content is nested between a component's opening and closing tags. For example, <Card><h2>Title</h2></Card> — inside the Card component, props.children would be the <h2>Title</h2> element. This enables the "composition" pattern, where a wrapper component (like a Modal, Card, or Layout) defines structure/styling and lets the calling code inject any custom content inside, rather than the wrapper needing dozens of props to handle every possible content variation.
  • Q6. What is the difference between a React Fragment and a div wrapper?
    JSX requires a component to return a single root element. If you don't actually want an extra wrapping element in the rendered DOM (for example, it would break your CSS grid/flex layout), you use a Fragment (<>...</> or <React.Fragment>...</React.Fragment>) instead of a <div>. A Fragment groups multiple JSX elements to satisfy the "single root" rule WITHOUT adding any actual node to the rendered HTML output, while a <div> wrapper does add an extra DOM element which could affect styling or semantics.
  • Q7. How would you conditionally render a component based on a boolean value?
    Several patterns: (1) Logical && operator for "render only if true": {isLoggedIn && <Dashboard />} — but be careful, if the left side evaluates to 0 (a falsy but renderable value), React will literally render "0" on screen, so prefer explicit boolean checks. (2) Ternary operator for "render A or B": {isLoggedIn ? <Dashboard /> : <LoginForm />}. (3) Early return with `if (!data) return null;` before the main JSX for "render nothing" cases. (4) if/else statements assigning to a variable before the return, for more complex branching logic.
  • Q8. What is the Virtual DOM and why does React use it?
    The Virtual DOM is a lightweight, in-memory JavaScript representation of the actual browser DOM. When a component's data changes, React creates a new Virtual DOM tree reflecting the updated UI, compares ("diffs") it against the previous Virtual DOM tree to find exactly what changed, then applies only those minimal, specific changes to the real DOM. Directly manipulating the real DOM is comparatively slow because it triggers browser reflows/repaints; by batching and minimizing actual DOM operations through this diffing process, React achieves much better performance than naive re-rendering of the entire page on every change.
