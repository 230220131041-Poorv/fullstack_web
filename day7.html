What is Git? What is GitHub?
Two different things people often confuse — know the distinction

Git vs GitHub — Not the Same Thing

Git is a distributed version control system — a tool installed on your computer that tracks every change made to your files over time. It runs locally and works completely offline. Created by Linus Torvalds (creator of Linux) in 2005.

GitHub is a cloud hosting platform for Git repositories — a website where you store your Git repos online, collaborate with teams, do code reviews via Pull Requests, and showcase your portfolio. GitHub is a company/product that USES Git; it didn't invent it. Other alternatives include GitLab and Bitbucket.

Analogy: Git is like Microsoft Word's "Track Changes" feature, but supercharged — it remembers every single saved version of your entire project forever, and lets multiple people work on different parts simultaneously without overwriting each other. GitHub is like Google Drive — a place to store that Word document online and share it with collaborators.

Why Version Control is Non-Negotiable

Without Git: developers email zip files back and forth, overwrite each other's work, and have no way to undo a bad change from 3 weeks ago. With Git: every change is tracked, multiple developers work in parallel via branches, mistakes are reversible, and you have a complete history of WHO changed WHAT and WHY (via commit messages).

Setup & Configuration
One-time setup before you start using Git
Bash — Git Configuration
# Check if Git is installed
git --version

# Set your identity — used in every commit you make
git config --global user.name "Poorv Patel"
git config --global user.email "poorv@email.com"

# Set default branch name to 'main' (modern standard)
git config --global init.defaultBranch main

# Set default editor for commit messages
git config --global core.editor "code --wait"  # VS Code

# View all your config settings
git config --list

# Set up SSH key for GitHub (passwordless push/pull) — one time setup
ssh-keygen -t ed25519 -C "poorv@email.com"
# Then add the public key (id_ed25519.pub) to GitHub → Settings → SSH Keys

Bash — Basic Workflow Commands
# ── INITIALIZE a new Git repository ──
git init                       # creates a hidden .git folder — starts tracking this directory

# ── CLONE an existing repository (copy from GitHub to your computer) ──
git clone https://github.com/user/repo.git
git clone git@github.com:user/repo.git  # SSH version

# ── CHECK STATUS — your most-used command, run it constantly ──
git status                     # shows modified/staged/untracked files

# ── STAGE changes (move to staging area) ──
git add index.html              # stage ONE specific file
git add src/                    # stage an entire folder
git add .                       # stage ALL changed files in current directory
git add -A                     # stage ALL changes in the whole repo

# ── COMMIT (save a checkpoint with a message) ──
git commit -m "Add navbar component"     # -m = inline message
git commit -am "Fix login bug"            # -a stages tracked files + commits in one step

# ── VIEW HISTORY ──
git log                         # full commit history
git log --oneline             # condensed: one line per commit
git log --oneline --graph --all  # visual branch graph
git log -p                    # show actual diff content per commit

# ── SEE WHAT CHANGED ──
git diff                        # unstaged changes vs last commit
git diff --staged             # staged changes vs last commit
git diff main feature-branch     # compare two branches

# ── PUSH & PULL with remote (GitHub) ──
git push origin main            # upload local commits to GitHub
git pull origin main            # download + merge remote changes into local
git fetch origin                # download remote changes WITHOUT merging (safer, inspect first)

Writing Good Commit Messages

Commit messages are documentation. A good commit message follows this pattern: short summary (50 chars) in imperative mood, blank line, then details if needed.

Good vs Bad Commit Messages
## ❌ BAD commit messages:
# "fix"
# "update stuff"
# "asdf"
# "final version"
# "WIP"

## ✅ GOOD commit messages (imperative mood, descriptive):
# "Add user authentication with JWT"
# "Fix navbar overlap on mobile screens"
# "Refactor product list to use useMemo for performance"
# "Remove unused lodash dependency"

# Conventional Commits format (used by many companies):
# feat: add shopping cart persistence
# fix: resolve race condition in checkout
# docs: update README installation steps
# style: format code with prettier
# refactor: extract validation logic to separate module
# test: add unit tests for cart total calculation
# chore: update dependencies

Branching
Working on features in parallel without breaking the main codebase

Why Branches Exist

A branch is an independent line of development. The default branch is usually called main (or historically master) and should always contain stable, working code. When building a new feature or fixing a bug, you create a separate branch, make your changes there, and only merge back into main once it's tested and reviewed.

Why this matters at companies: Multiple developers work simultaneously without stepping on each other's code. If a feature branch breaks something, main stays unaffected. Code review happens on the branch before it's allowed into main.
main ●───●───●───────────●───● │ ╲ ╱ feature └──●───●───●
main stays stable while feature branch develops independently, then merges back in once ready.
Bash — Branching Commands
# ── VIEWING BRANCHES ──
git branch                     # list local branches (* marks current)
git branch -a                  # list ALL branches (local + remote)
git branch -v                  # list with last commit info

# ── CREATING & SWITCHING BRANCHES ──
git branch feature-login         # create a new branch (doesn't switch to it)
git checkout feature-login       # switch to that branch (older command)
git switch feature-login         # switch to that branch (modern command)

# Create AND switch in one command (most commonly used)
git checkout -b feature-login    # older syntax
git switch -c feature-login      # modern syntax

# ── NAMING CONVENTIONS used at companies ──
# feature/user-authentication
# bugfix/navbar-overlap
# hotfix/payment-crash
# release/v2.0
# chore/update-dependencies

# ── RENAMING a branch ──
git branch -m old-name new-name

# ── DELETING branches ──
git branch -d feature-login       # delete (safe — warns if unmerged)
git branch -D feature-login       # force delete (even if unmerged — careful!)
git push origin --delete feature-login  # delete branch on GitHub too

# ── PUSHING a new branch to GitHub for the first time ──
git push -u origin feature-login  # -u sets upstream tracking (only needed once per branch)
git push                          # after -u is set, just 'git push' works

Merging & Resolving Conflicts
Bringing branches back together — and handling the inevitable conflicts
Bash — Merging
# ── MERGING a feature branch into main ──
git switch main                  # 1. go to the branch you want to merge INTO
git pull origin main             # 2. make sure main is up to date
git merge feature-login          # 3. merge feature-login INTO main
git push origin main             # 4. push the merged result

# ── TYPES OF MERGES ──
# Fast-forward merge: happens when main has had NO new commits
#   since the feature branch was created — Git just moves the pointer forward.

# 3-way merge: happens when BOTH branches have new commits —
#   Git creates a new "merge commit" that combines both histories.
git merge --no-ff feature-login  # force a merge commit even if fast-forward possible
                                  # (keeps feature history visible — many teams prefer this)

What Causes a Merge Conflict?

A conflict happens when Git can't automatically decide how to combine changes — typically when the same lines in the same file were changed differently on two branches. Git pauses the merge and asks YOU to decide which version (or combination) is correct.

Bash — Resolving a Merge Conflict
# When a conflict happens, Git marks the file like this:

/* In the conflicted file: */
<<<<<<< HEAD
// your current branch's version
const apiUrl = 'https://api-v2.example.com';
=======
// the incoming branch's version
const apiUrl = 'https://api.example.com/v2';
>>>>>>> feature-login

# ── STEPS TO RESOLVE ──
# 1. Open the conflicted file in your editor (VS Code highlights conflicts nicely)
# 2. Manually edit — choose ONE version, BOTH combined, or something new
# 3. Remove the <<<<<<<, =======, >>>>>>> markers completely
# 4. Save the file, then:

git add conflicted-file.js       # mark as resolved
git commit -m "Resolve merge conflict in api config"

# ── IF YOU WANT TO ABORT a messy merge and start over ──
git merge --abort             # cancels the merge, returns to pre-merge state

# ── TOOLS TO HELP ──
git status                     # shows which files have conflicts
# VS Code shows "Accept Current", "Accept Incoming", "Accept Both" buttons inline
🏢 Real World

Merge Conflicts in Team Projects

Merge conflicts are completely normal and happen constantly on real teams — they are NOT a sign you did something wrong. They simply mean two people edited the same part of a file. Interviewers want to see that you stay calm and know the resolution process, not that you avoid conflicts entirely.

The Pull Request (PR) Workflow — How Real Teams Collaborate

A Pull Request is a GitHub feature (not a Git command) that proposes merging your branch into another branch, with a review step before it's allowed in. This is THE standard workflow at every tech company.

Standard Company Workflow (Step by Step)

  1. Pull latest main to stay updated: git pull origin main
  2. Create a feature branch: git switch -c feature/add-cart
  3. Make changes, commit frequently with clear messages
  4. Push the branch to GitHub: git push -u origin feature/add-cart
  5. Open a Pull Request on GitHub comparing your branch → main
  6. Teammates review the code, leave comments, request changes
  7. You push more commits to address feedback (PR auto-updates)
  8. Once approved, the PR is merged (often "squash and merge" to keep history clean)
  9. The feature branch is deleted; everyone pulls the updated main
Fork vs Clone

Clone: downloads a copy of a repo you have access to, linked to the original.

Fork: creates your OWN copy of someone else's repo under your GitHub account — used for open-source contributions where you don't have direct write access.

Issues

GitHub's built-in tracker for bugs, feature requests, and tasks. Teams link Pull Requests to Issues (e.g., "Fixes #42") so merging a PR can auto-close the related issue.

Undoing Changes
Mistakes happen — knowing how to safely undo them is a critical skill
Bash — Undoing Changes (most asked Git topic!)
# ── UNDOING UNSTAGED changes (working directory) ──
git checkout -- file.js        # discard changes in ONE file (back to last commit)
git restore file.js            # modern equivalent of above
git restore .                  # discard ALL unstaged changes — ⚠️ DESTRUCTIVE

# ── UNSTAGING (move from staging area back to working directory) ──
git reset file.js               # unstage ONE file (keeps your edits, just unstages)
git restore --staged file.js    # modern equivalent

# ── UNDOING COMMITS ──
git commit --amend -m "New message"  # fix the LAST commit's message
git commit --amend --no-edit       # add staged changes to last commit, keep same message

# git reset: moves the branch pointer back, 3 modes:
git reset --soft  HEAD~1       # undo last commit, KEEP changes staged
git reset --mixed HEAD~1       # undo last commit, KEEP changes unstaged (default mode)
git reset --hard  HEAD~1       # undo last commit, DELETE changes entirely ⚠️ DANGEROUS

# git revert: SAFER than reset — creates a NEW commit that undoes a previous one
# (doesn't rewrite history — safe for commits already pushed/shared)
git revert a1b2c3d             # creates a new "revert commit" undoing that specific commit
git revert HEAD               # revert the most recent commit

# ── VIEWING OLD VERSIONS ──
git log --oneline             # find the commit hash you want
git show a1b2c3d             # view that specific commit's changes
git checkout a1b2c3d -- file.js  # restore ONE file from an old commit

# ── RECOVERING "lost" commits (reflog is a lifesaver) ──
git reflog                     # shows EVERY action — even after reset --hard, commits aren't gone yet
git reset --hard a1b2c3d      # jump back to a commit found in reflog
Reset vs Revert — the most common interview question on this topic:
git reset rewrites history — moves the branch pointer backward, as if the commit never happened. Dangerous on shared/pushed branches because it breaks history for anyone who already pulled.

git revert is non-destructive — it creates a NEW commit that undoes the changes, keeping full history intact. ALWAYS use revert on commits that have already been pushed/shared with others.

Stash, Rebase & Cherry-pick
Advanced commands used by experienced developers
Bash — Stash (temporary shelf for changes)
# Scenario: you're mid-work on a feature, but need to urgently switch
# branches to fix something — but your changes aren't ready to commit yet.

git stash                      # shelves all uncommitted changes, gives you a clean working directory
git stash save "WIP: cart logic"  # stash with a descriptive message
git switch main                # now you can safely switch branches
# ... fix the urgent bug, commit, push ...
git switch feature-branch       # go back to your feature branch
git stash pop                  # restore your shelved changes AND remove from stash list
git stash apply                # restore changes but KEEP them in stash list too

git stash list                 # see all stashed changes
git stash drop                 # delete the most recent stash
git stash clear                # delete ALL stashes
Bash — Rebase (rewriting commit history)
# Rebase replays your branch's commits ON TOP of another branch's latest commits
# — creates a CLEAN, LINEAR history instead of a merge commit.

git switch feature-branch
git rebase main                 # replay feature-branch's commits on top of latest main

# Interactive rebase — edit, reorder, squash, or delete recent commits
git rebase -i HEAD~3             # interactively edit last 3 commits
# Opens an editor with options per commit:
#   pick   = keep commit as is
#   squash = combine with previous commit (merges messages)
#   reword = keep changes, edit just the message
#   drop   = delete this commit entirely

# ── MERGE vs REBASE — the eternal debate ──
# merge: preserves exact history, creates a merge commit, safe for shared branches
# rebase: rewrites history into a clean line, NEVER rebase a branch others
#         are also working on (rewrites commit hashes — breaks their history)

# Golden rule: "Never rebase what you've already pushed and shared"
Bash — Cherry-pick
# Apply ONE specific commit from another branch, without merging the whole branch
git cherry-pick a1b2c3d         # grabs just that one commit's changes onto current branch

# Real use case: a critical bugfix was committed on a feature branch,
# but you need it on main IMMEDIATELY without merging the whole
# (possibly unfinished) feature branch.

.gitignore & Best Practices
Keeping your repo clean — what should NEVER be committed

What is .gitignore?

A .gitignore file tells Git which files/folders to NEVER track — things like dependencies, build outputs, environment secrets, and OS-specific files. Without it, you'd accidentally commit thousands of unnecessary files (or worse, leak secrets).

.gitignore — Standard Full Stack Project
# Dependencies — NEVER commit these, they're huge and regenerable via package.json
node_modules/
# Environment variables — contains API keys, DB passwords, SECRETS!
.env
.env.local
.env.production

# Build output — generated files, not source code
dist/
build/
.next/
out/

# Logs
*.log
npm-debug.log*

# OS-specific files
.DS_Store        # macOS
Thumbs.db        # Windows

# Editor folders
.vscode/
.idea/

# Test coverage reports
coverage/

# Cache
.cache/
.eslintcache
⚠️ Critical: NEVER commit .env files or API keys! If you accidentally commit secrets, simply deleting the file later isn't enough — it remains in Git history forever (recoverable via git log). You must rotate/regenerate the leaked keys immediately. Always add .env to .gitignore BEFORE your first commit.

General Best Practices for Placements

  • Commit often, in small logical chunks — not one giant "final commit"
  • Write clear, descriptive commit messages (imperative mood: "Add", "Fix", "Update")
  • Always pull before you start working to avoid unnecessary conflicts
  • Never force-push (git push --force) to shared branches like main
  • Keep a clean, well-documented README.md in every portfolio project
  • Use branches for every feature, never commit directly to main on team projects

Placement Interview Q&A
Top Git & GitHub questions from real company interviews
  • Q1. What is the difference between Git and GitHub?
    Git is a distributed version control system — software installed locally that tracks changes to files over time, works fully offline, and was created by Linus Torvalds. GitHub is a cloud-based hosting platform for Git repositories, owned by Microsoft, that adds collaboration features on top of Git: Pull Requests for code review, Issues for tracking bugs/tasks, Actions for CI/CD, and a social layer for open source. You could use Git without ever touching GitHub (e.g., with a local repo or a different host like GitLab).
  • Q2. What is the difference between git merge and git rebase?
    Both integrate changes from one branch into another, but differently. git merge creates a new "merge commit" that ties the two histories together, preserving exactly what happened and when — safe for shared/public branches. git rebase replays your branch's commits on top of the target branch's latest commit, creating a clean, linear history with no merge commit — but it REWRITES commit hashes, which is dangerous if others have already pulled and based work on those commits. Golden rule: never rebase a branch that others are also working on or have already pulled.
  • Q3. What is the difference between git reset and git revert?
    git reset moves the branch pointer backward, effectively erasing commits from history (--soft keeps changes staged, --mixed keeps them unstaged, --hard deletes them entirely). It rewrites history, which is dangerous for commits already pushed and shared with others. git revert creates a brand NEW commit that undoes the changes from a previous commit, without altering history — the original commit still exists, but its effects are cancelled out by the new revert commit. Always use revert for commits that have already been pushed to a shared branch; reset is fine for local, unpushed commits.
  • Q4. How do you resolve a merge conflict?
    When Git can't auto-merge because the same lines changed differently on both branches, it pauses and marks the conflicted sections in the file with <<<<<<< HEAD, =======, and >>>>>>> branch-name markers. You open the file, manually decide which version to keep (or combine both), delete the conflict markers, save the file, then run git add on the resolved file and git commit to complete the merge. Tools like VS Code show "Accept Current/Incoming/Both" buttons to simplify this. If things get too messy, git merge --abort cancels the merge and returns to the pre-merge state.
  • Q5. What is git stash used for?
    git stash temporarily shelves uncommitted changes (both staged and unstaged) without committing them, giving you a clean working directory. This is useful when you need to quickly switch branches — for example, to fix an urgent bug on main — but your current work isn't ready to be committed yet. After finishing the urgent task, you switch back to your original branch and run git stash pop to restore your shelved changes exactly where you left off.
  • Q6. What is a Pull Request and why is it important?
    A Pull Request (PR) is a GitHub feature (not a core Git command) that proposes merging changes from one branch into another, typically a feature branch into main. It's the standard mechanism for code review in professional teams: teammates can see the diff, leave inline comments, request changes, and approve before the code is allowed to merge. PRs create accountability, catch bugs/style issues before they reach production, and document why changes were made (via the PR description and discussion).
  • Q7. What should you NEVER commit to a Git repository?
    Never commit: .env files or any file containing API keys, passwords, or secrets; the node_modules folder (huge, regenerable from package.json via npm install); build/compiled output folders (dist, build); OS-specific files (.DS_Store, Thumbs.db); and IDE-specific config folders unless team-shared. These should all be listed in .gitignore from the very first commit. If secrets are accidentally committed, removing the file later isn't enough — it remains in Git history and must be purged with tools like git filter-branch or BFG Repo-Cleaner, and the leaked credentials must be rotated immediately.
  • Q8. What's the difference between git fetch and git pull?
    git fetch downloads commits, branches, and files from the remote repository, but does NOT merge them into your current branch — it just updates your local knowledge of what's on the remote, letting you inspect changes before deciding to integrate them. git pull is essentially git fetch followed immediately by git merge (or rebase, if configured) — it downloads AND automatically merges remote changes into your current branch in one step. Fetch is the "safer, look before you leap" option; pull is the convenient, common everyday command.
