The Prop Drilling Problem
Why Context exists — the pain it solves

What is Prop Drilling?

Prop drilling happens when you need to pass data through many layers of components, even though only the deepest component actually uses it. Every component in between must accept and forward the prop just to relay it, even if they don't need it themselves.

Real-world example: A logged-in user's theme preference needs to reach a deeply nested Button component. You'd have to pass "theme" through App → Layout → Sidebar → Menu → MenuItem → Button — 5 components just relaying a prop they never use themselves.
JSX — Prop Drilling Example (the problem)
function App() {
  const [theme, setTheme] = useState('dark');
  return <Layout theme={theme} />;  // pass it down...
}

function Layout({ theme }) {
  return <Sidebar theme={theme} />;  // Layout doesn't use theme, just relays it
}

function Sidebar({ theme }) {
  return <Menu theme={theme} />;    // Sidebar doesn't use theme either...
}

function Menu({ theme }) {
  return <MenuItem theme={theme} />; // still just relaying...
}

function MenuItem({ theme }) {
  return <button className={theme}>Click</button>; // FINALLY used here
}

// 4 components had to know about "theme" just to pass it along.
// Adding a new layer, renaming the prop, or adding more shared data
// means touching every single file in the chain. This doesn't scale.
🌐
Context API Basics
Built-in React solution — share state without passing through every layer

What is Context?

The Context API lets you create a "global" value for a part of your component tree, which ANY descendant can read directly — no matter how deeply nested — without it being passed as a prop through every intermediate component. There are 3 steps: create the context, provide a value, and consume it wherever needed.

JSX — Solving Prop Drilling with Context
import { createContext, useContext, useState } from 'react';

// ── STEP 1: CREATE the context (usually in its own file) ──
const ThemeContext = createContext();  // optionally pass a default value: createContext('light')

// ── STEP 2: PROVIDE a value at the top of the tree ──
function App() {
  const [theme, setTheme] = useState('dark');

  return (
    <ThemeContext.Provider value={{ theme, setTheme }}>
      <Layout />   // theme is now available to EVERY descendant, no props needed
    </ThemeContext.Provider>
  );
}

// ── Intermediate components — NO LONGER need to know about theme at all! ──
function Layout() {
  return <Sidebar />;  // clean — no theme prop being relayed
}
function Sidebar() {
  return <Menu />;
}
function Menu() {
  return <MenuItem />;
}

// ── STEP 3: CONSUME the context wherever it's actually needed ──
function MenuItem() {
  const { theme, setTheme } = useContext(ThemeContext); // reaches DIRECTLY into context

  return (
    <button
      className={theme}
      onClick={() => setTheme(theme === 'dark' ? 'light' : 'dark')}
    >
      Toggle Theme
    </button>
  );
}
// Only the components that ACTUALLY need theme touch the Context — everyone else stays clean
How Context re-rendering works: Whenever the value passed to a Provider changes, ALL components consuming that context (via useContext) re-render — regardless of how deep they are. This is powerful but means you shouldn't put rapidly-changing or unrelated data into one giant context, as it can cause unnecessary re-renders across your app.

Custom Hooks + Context Pattern
The professional way real codebases structure Context

Wrapping Context in a Custom Hook

Real production codebases almost never call useContext directly in components. Instead, they wrap it in a custom hook (a function starting with use that uses other hooks inside). This gives cleaner imports and lets you add error handling for misuse.

JSX — Full Auth Context Pattern (Real-World Structure)
// ── AuthContext.jsx — a complete, reusable context module ──
import { createContext, useContext, useState, useEffect } from 'react';

const AuthContext = createContext(null);

export function AuthProvider({ children }) {
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    // check if user is already logged in (e.g. from a saved token)
    const savedUser = localStorage.getItem('user');
    if (savedUser) setUser(JSON.parse(savedUser));
    setLoading(false);
  }, []);

  async function login(email, password) {
    const response = await fetch('/api/login', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ email, password })
    });
    const data = await response.json();
    setUser(data.user);
    localStorage.setItem('user', JSON.stringify(data.user));
  }

  function logout() {
    setUser(null);
    localStorage.removeItem('user');
  }

  const value = { user, loading, login, logout, isAuthenticated: !!user };

  return <AuthContext.Provider value={value}>{children}</AuthContext.Provider>;
}

// ── CUSTOM HOOK — the clean public interface to this context ──
export function useAuth() {
  const context = useContext(AuthContext);
  if (context === null) {
    throw new Error('useAuth must be used within an AuthProvider'); // catches misuse early!
  }
  return context;
}

// ── main.jsx — wrap the WHOLE app once ──
import { AuthProvider } from './AuthContext';
createRoot(document.getElementById('root')).render(
  <AuthProvider>
    <App />
  </AuthProvider>
);

// ── ANY component, anywhere, can now use it cleanly ──
import { useAuth } from './AuthContext';

function ProfileButton() {
  const { user, logout, isAuthenticated } = useAuth(); // one clean line, no useContext/import of raw Context

  if (!isAuthenticated) return <a href="/login">Log In</a>;

  return (
    <div>
      <span>{user.name}</span>
      <button onClick={logout}>Log Out</button>
    </div>
  );
}

Context API vs Redux / Zustand
Knowing when built-in Context is enough — vs when you need a state library
Use Context for…

Theme (dark/light mode), authenticated user info, language/locale preference, simple global UI state (modal open/closed). Data that changes infrequently and isn't extremely performance-sensitive.

Use Redux/Zustand for…

Large apps with complex, frequently-changing state shared across many unrelated parts of the tree (e.g., a shopping cart with live price calculations, real-time collaborative data). These libraries offer better performance optimization and devtools for large-scale state.

Interview framing: Context API is built into React (no extra dependency) and perfect for simple, infrequently-changing global state. Redux/Zustand/Recoil are external libraries built for complex state logic, offering finer-grained re-render control, middleware, and debugging tools — but add complexity. Most placement-level projects only need Context.

React Router — Introduction
Turning a Single Page App into a multi-page experience

What Problem Does React Router Solve?

React apps are Single Page Applications (SPAs) — only ONE actual HTML page is ever loaded; everything else is JavaScript swapping out content in the DOM. Without a router, you can't have shareable URLs like /products/42 or /about — there's nothing connecting URLs to which component should render.

React Router (the most widely used routing library, now on v6+) maps URL paths to components, handles browser back/forward buttons, and updates the URL without a full page reload — giving you the UX of a traditional multi-page site with the performance and smoothness of an SPA.

Bash — Installing React Router
npm install react-router-dom
⚙️
Router Setup & Basic Routes
BrowserRouter, Routes, and Route — the foundation
JSX — Basic Router Setup
import { BrowserRouter, Routes, Route } from 'react-router-dom';
import Home from './pages/Home';
import About from './pages/About';
import Products from './pages/Products';
import Contact from './pages/Contact';
import NotFound from './pages/NotFound';

function App() {
  return (
    <BrowserRouter>            {/* wraps the WHOLE app — enables routing */}
      <Routes>                  {/* container for all your Route definitions */}
        <Route path="/" element={<Home />} />
        <Route path="/about" element={<About />} />
        <Route path="/products" element={<Products />} />
        <Route path="/contact" element={<Contact />} />
        <Route path="*" element={<NotFound />} />  {/* catch-all 404 — always LAST */}
      </Routes>
    </BrowserRouter>
  );
}

// Only ONE Route's element renders at a time, matching the current URL path.
// React Router compares the browser's current path against each "path" prop.

Navigation: Link, NavLink, useNavigate
Moving between pages without full reloads

Why Not Just Use <a href>?

A regular <a href="/about"> tag triggers a FULL PAGE RELOAD — the browser re-downloads the entire app, defeating the purpose of an SPA. React Router's <Link> intercepts the click and updates the URL + renders the new component WITHOUT reloading the page.

JSX — Navigation Components
import { Link, NavLink, useNavigate } from 'react-router-dom';

// ── Link — basic client-side navigation (replaces <a>) ──
function Navbar() {
  return (
    <nav>
      <Link to="/">Home</Link>
      <Link to="/products">Products</Link>
      <Link to="/about">About</Link>
    </nav>
  );
}

// ── NavLink — like Link, but knows when it's "active" (current page) ──
function Navbar() {
  return (
    <nav>
      <NavLink
        to="/products"
        className={({ isActive }) => isActive ? 'nav-link active' : 'nav-link'}
      >
        Products
      </NavLink>
    </nav>
  );
}
// NavLink auto-applies styling/class when its 'to' matches the current URL — great for highlighting the active nav item

// ── useNavigate — programmatic navigation (e.g. after form submit) ──
function LoginForm() {
  const navigate = useNavigate();

  async function handleSubmit(e) {
    e.preventDefault();
    const success = await loginUser();
    if (success) {
      navigate('/dashboard');       // redirect after successful login
    }
  }

  function goBack() {
    navigate(-1);  // go back one step in history (like browser back button)
  }

  return <form onSubmit={handleSubmit}>...</form>;
}

Dynamic Routes & URL Parameters
One route template handling infinite pages — product detail, user profile, etc.

The Problem Dynamic Routes Solve

You don't want to write a separate <Route> for every single product (/products/1, /products/2, ...). Instead, you define ONE route with a dynamic parameter using a colon, and read the actual value from the URL using useParams.

JSX — Dynamic Routes with useParams
import { Routes, Route, useParams, useSearchParams } from 'react-router-dom';

// ── DEFINING a dynamic route ──
<Routes>
  <Route path="/products/:productId" element={<ProductDetail />} />
  <Route path="/users/:userId/posts/:postId" element={<UserPost />} />  {/* multiple params */}
</Routes>

// ── READING the param inside the component ──
function ProductDetail() {
  const { productId } = useParams();  // extracts ':productId' from current URL — always a STRING
  const [product, setProduct] = useState(null);

  useEffect(() => {
    fetch(`/api/products/${productId}`)
      .then(res => res.json())
      .then(setProduct);
  }, [productId]); // re-fetch if the user navigates to a DIFFERENT product

  if (!product) return <p>Loading...</p>;

  return (
    <div>
      <h1>{product.name}</h1>
      <p>₹{product.price}</p>
    </div>
  );
}

// ── QUERY STRINGS (?key=value) with useSearchParams ──
// URL example: /products?category=electronics&sort=price
function ProductList() {
  const [searchParams, setSearchParams] = useSearchParams();

  const category = searchParams.get('category'); // 'electronics'
  const sort     = searchParams.get('sort');     // 'price'

  function updateCategory(newCategory) {
    setSearchParams({ category: newCategory, sort }); // updates the URL ?query string
  }

  return <div>Filtering by: {category}, sorted by: {sort}</div>;
}
Gotcha: URL params from useParams() are ALWAYS strings, even if they look numeric. const { productId } = useParams() gives you the string "42", not the number 42. Convert with Number(productId) if you need to do math or strict-equality compare with numeric IDs.

Nested Routes & Shared Layouts
Reusing a navbar/sidebar across multiple pages without repeating it

The Outlet Component

When multiple pages share a common layout (navbar, sidebar, footer), you define a parent "layout route" and use <Outlet /> as a placeholder for wherever the matching CHILD route's content should render.

JSX — Nested Routes with Outlet
import { Routes, Route, Outlet, Link } from 'react-router-dom';

// ── DashboardLayout.jsx — shared shell for all dashboard pages ──
function DashboardLayout() {
  return (
    <div className="dashboard">
      <nav className="sidebar">
        <Link to="/dashboard">Overview</Link>
        <Link to="/dashboard/orders">Orders</Link>
        <Link to="/dashboard/settings">Settings</Link>
      </nav>
      <main>
        <Outlet />  {/* the matching child route renders HERE */}
      </main>
    </div>
  );
}

// ── App.jsx — defining the route nesting ──
<Routes>
  <Route path="/dashboard" element={<DashboardLayout />}>
    <Route index element={<Overview />} />          {/* renders at exactly /dashboard */}
    <Route path="orders" element={<Orders />} />    {/* /dashboard/orders */}
    <Route path="settings" element={<Settings />} /> {/* /dashboard/settings */}
  </Route>
</Routes>
// The sidebar nav renders ONCE; only <Outlet/>'s content swaps as you navigate
// between Overview / Orders / Settings — no repeated layout code!

Protected Routes
Blocking access to pages unless the user is authenticated
JSX — Protected Route Pattern (combines Context + Router)
import { Navigate, Outlet } from 'react-router-dom';
import { useAuth } from './AuthContext';

// ── A wrapper component that guards its children ──
function ProtectedRoute() {
  const { isAuthenticated, loading } = useAuth();

  if (loading) return <p>Loading...</p>;  // wait for auth check to complete

  if (!isAuthenticated) {
    return <Navigate to="/login" replace />;  // redirect, don't render children
  }

  return <Outlet />;  // authenticated — render whatever child route matched
}

// ── USAGE: wrap any routes that require login ──
<Routes>
  <Route path="/login" element={<Login />} />
  <Route path="/" element={<Home />} />

  <Route element={<ProtectedRoute />}>     {/* guard applies to ALL routes nested below */}
    <Route path="/dashboard" element={<Dashboard />} />
    <Route path="/settings" element={<Settings />} />
  </Route>
</Routes>
// Trying to visit /dashboard while logged out → instantly redirected to /login

Real Project: E-commerce App Shell
Combining Context (cart state) + Router (multi-page navigation)
🏢 Real World

This exact combination powers most e-commerce sites

Cart state needs to be available on every page (Context), while Home, Product Detail, Cart, and Checkout are separate routes (Router) — this is the standard architecture for shopping platforms built by Ahmedabad web development companies.

JSX — Complete App Structure
// ── CartContext.jsx ──
import { createContext, useContext, useState } from 'react';

const CartContext = createContext(null);

export function CartProvider({ children }) {
  const [items, setItems] = useState([]);

  function addToCart(product) {
    setItems(prev => [...prev, product]);
  }
  function removeFromCart(productId) {
    setItems(prev => prev.filter(p => p.id !== productId));
  }
  const cartCount = items.length;
  const cartTotal = items.reduce((sum, p) => sum + p.price, 0);

  return (
    <CartContext.Provider value={{ items, addToCart, removeFromCart, cartCount, cartTotal }}>
      {children}
    </CartContext.Provider>
  );
}

export function useCart() {
  return useContext(CartContext);
}

// ── App.jsx — Router + Context combined ──
import { BrowserRouter, Routes, Route } from 'react-router-dom';
import { CartProvider } from './CartContext';
import Layout from './Layout';
import Home from './pages/Home';
import ProductDetail from './pages/ProductDetail';
import Cart from './pages/Cart';

function App() {
  return (
    <CartProvider>                  {/* cart state available EVERYWHERE inside */}
      <BrowserRouter>
        <Routes>
          <Route path="/" element={<Layout />}>
            <Route index element={<Home />} />
            <Route path="products/:productId" element={<ProductDetail />} />
            <Route path="cart" element={<Cart />} />
          </Route>
        </Routes>
      </BrowserRouter>
    </CartProvider>
  );
}

// ── Layout.jsx — navbar shows live cart count on EVERY page ──
import { Outlet, Link } from 'react-router-dom';
import { useCart } from './CartContext';

function Layout() {
  const { cartCount } = useCart();  // no prop drilling — reads cart directly!

  return (
    <div>
      <nav>
        <Link to="/">Shop</Link>
        <Link to="/cart">Cart ({cartCount})</Link>
      </nav>
      <Outlet />
    </div>
  );
}

// ── ProductDetail.jsx — uses BOTH Router (params) and Context (cart) ──
import { useParams } from 'react-router-dom';
import { useCart } from './CartContext';

function ProductDetail() {
  const { productId } = useParams();    // from the URL
  const { addToCart } = useCart();      // from Context
  const [product, setProduct] = useState(null);

  useEffect(() => {
    fetch(`/api/products/${productId}`).then(r => r.json()).then(setProduct);
  }, [productId]);

  if (!product) return <p>Loading...</p>;

  return (
    <div>
      <h1>{product.name}</h1>
      <button onClick={() => addToCart(product)}>Add to Cart</button>
    </div>
  );
}

Placement Interview Q&A
Top Context API & React Router questions for company interviews
  • Q1. What is prop drilling and how does Context solve it?
    Prop drilling is passing a prop through multiple layers of components that don't actually use it themselves, just to get it to a deeply nested component that does. This makes refactoring painful — every intermediate component must know about and forward the prop. Context API solves this by letting you create a value at a high level (via a Provider) that any descendant component can access directly with useContext, regardless of nesting depth, without it being threaded through every component in between.
  • Q2. Explain the 3 steps to use Context API.
    (1) Create the context with createContext(), typically with a default value or null. (2) Provide a value by wrapping part of your component tree in — usually near the top of the app or around the section that needs the shared data. (3) Consume the context in any descendant component using the useContext(Context) hook, which returns whatever value the nearest Provider above it supplied.
  • Q3. Why do real codebases wrap useContext in a custom hook instead of calling it directly?
    Wrapping it in a custom hook (e.g., useAuth() instead of useContext(AuthContext)) gives a cleaner, more readable API for consumers — they don't need to import and know about the raw Context object. It also lets you add a safety check that throws a clear error if the hook is used outside its Provider (e.g., "useAuth must be used within an AuthProvider"), catching integration mistakes early with a helpful message instead of a confusing undefined error deep in the component.
  • Q4. What is React Router and what problem does it solve in a Single Page Application?
    React apps are Single Page Applications — only one HTML file is ever loaded, and JavaScript swaps content in and out of the DOM. Without a router, there's no concept of distinct, shareable, bookmarkable URLs (like /products/42) mapping to different views, and the browser back/forward buttons wouldn't work meaningfully. React Router maps URL paths to components, intercepts navigation to avoid full page reloads, and keeps the URL bar, browser history, and rendered UI all in sync — giving the experience of a multi-page site with SPA performance.
  • Q5. What is the difference between Link and a regular anchor tag (a href)?
    A regular tag triggers a full page reload — the browser discards the current JavaScript app state and re-downloads everything from the server, which is slow and defeats the purpose of a SPA. React Router's intercepts the click with JavaScript, updates the URL and browser history, and re-renders only the matching Route's component — no network round-trip or full reload, preserving any existing app state (like Context values) that doesn't need to change.
  • Q6. How do you create a dynamic route and access its parameter?
    Define the route with a colon-prefixed segment, like } />. This matches any URL like /products/1, /products/42, etc. Inside the ProductDetail component, you call const { productId } = useParams() to extract the actual value from the current URL. Note that the value is always a string, even if it looks numeric — convert with Number(productId) if you need to do arithmetic or strict type comparisons.
  • Q7. What is the Outlet component used for?
    Outlet is a placeholder component used in nested routing — it marks WHERE a matching child route's element should render within a parent "layout route." For example, a DashboardLayout component can render a shared sidebar and header once, with an in the content area; as the user navigates between /dashboard, /dashboard/orders, and /dashboard/settings, only the content inside that Outlet swaps out, while the surrounding layout stays mounted and doesn't re-render unnecessarily.
  • Q8. How would you implement a protected route that requires authentication?
    Create a wrapper component (commonly called ProtectedRoute) that reads authentication state — usually from a Context like useAuth(). Inside, check if the user is authenticated: if not, return a component, which redirects without rendering the protected content at all. If authenticated, render an so the actual matched child route renders normally. You then wrap any routes requiring login as children of this ProtectedRoute in your route configuration, so the check applies uniformly to all of them.
