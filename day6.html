Synchronous vs Asynchronous
Why JavaScript needs async at all — the fundamental problem

JavaScript is Single-Threaded

JavaScript runs on a single thread — it can only do one thing at a time. In synchronous code, each line waits for the previous one to finish before running. This is fine for fast operations (math, string manipulation) but a disaster for slow operations like network requests, file reads, or timers — if these were synchronous, the entire page would freeze while waiting.

Asynchronous code lets slow operations run in the background without blocking the main thread. The rest of your code keeps running, and a callback/Promise/async function handles the result whenever it's ready.

Real-world analogy: Imagine ordering food at a restaurant. Synchronous = you stand at the counter and wait, blocking the entire queue, until your food is ready. Asynchronous = you place your order, get a token number, sit down, and the restaurant calls you when it's ready — meanwhile other people can also order.
JavaScript — Sync vs Async Comparison
// ── SYNCHRONOUS: blocks until done, runs top to bottom ──
console.log('1');
console.log('2');
console.log('3');
// Output: 1, 2, 3 — exactly in order, no surprises

// ── ASYNCHRONOUS: doesn't block, runs "later" ──
console.log('1');
setTimeout(() => console.log('2'), 0);  // scheduled for LATER, even with 0ms delay!
console.log('3');
// Output: 1, 3, 2  ← '2' runs AFTER all sync code finishes!
// This surprises every beginner — setTimeout doesn't pause execution.

// ── WHY THIS MATTERS: blocking operations would freeze the UI ──
// If fetch() were synchronous, clicking a button that loads data
// would freeze the ENTIRE page — no scrolling, no clicking anything —
// until the server responds (which could take seconds).
// Async lets the page stay responsive while waiting for the network.

The Event Loop
How JavaScript achieves async behaviour despite being single-threaded

The Mechanism Behind Async JS

JavaScript has one Call Stack (where code currently executes) but uses the Event Loop to manage async tasks via two queues: the Microtask Queue (Promises, async/await) and the Macrotask/Callback Queue (setTimeout, DOM events, I/O).

The Event Loop's job: when the Call Stack is empty, check the Microtask Queue first (drain it completely), then take ONE task from the Macrotask Queue, run it, and repeat.

JavaScript — Event Loop Execution Order (Classic Interview Question)
console.log('1: Sync start');

setTimeout(() => {
  console.log('2: setTimeout (macrotask)');
}, 0);

Promise.resolve().then(() => {
  console.log('3: Promise (microtask)');
});

console.log('4: Sync end');

// ── EXECUTION ORDER ──
// 1: Sync start
// 4: Sync end
// 3: Promise (microtask)
// 2: setTimeout (macrotask)

// WHY this order?
// Step 1: All SYNCHRONOUS code runs first, top to bottom (1, 4)
// Step 2: Call stack is empty → Event Loop checks MICROTASK queue first
//         → Promise callback runs (3)
// Step 3: Microtask queue empty → Event Loop takes ONE macrotask
//         → setTimeout callback runs (2)

// KEY RULE: Microtasks (Promises) ALWAYS run before Macrotasks (setTimeout),
// even if setTimeout has a delay of 0ms!
Why this is asked constantly in interviews: This single question tests whether you actually understand JS execution, not just syntax. Memorize: synchronous code first → all microtasks (Promises) → one macrotask (setTimeout) → repeat.

Callbacks & Callback Hell
The original way to handle async — and the problem it created

What is a Callback?

A callback is simply a function passed as an argument to another function, to be "called back" once an operation completes. Before Promises existed, this was the only way to handle async code — and it led to a famous problem called Callback Hell.

JavaScript — Callbacks & Callback Hell
// ── BASIC CALLBACK ──
function fetchUser(id, callback) {
  setTimeout(() => {              // simulating network delay
    callback({ id, name: 'Poorv' });
  }, 1000);
}

fetchUser(1, (user) => {
  console.log('Got user:', user);
});

// ── CALLBACK HELL — nested callbacks become unreadable ──
fetchUser(1, (user) => {
  fetchOrders(user.id, (orders) => {
    fetchOrderDetails(orders[0].id, (details) => {
      fetchShippingInfo(details.id, (shipping) => {
        console.log(shipping);
        // Nested 4 levels deep — "Pyramid of Doom"
        // Hard to read, hard to handle errors, hard to debug
      }, (err) => { console.error(err); });
    }, (err) => { console.error(err); });
  }, (err) => { console.error(err); });
}, (err) => { console.error(err); });

// Problems with callback hell:
// 1. Hard to read — deeply nested, indentation explodes
// 2. Error handling repeated at every level
// 3. Hard to run things in parallel
// 4. Hard to reason about execution order
// → This is EXACTLY why Promises were introduced

Promises
A cleaner way to represent the eventual result of an async operation

What is a Promise?

A Promise is an object representing the eventual completion (or failure) of an asynchronous operation. It's a placeholder for a value that isn't available yet but will be at some point. A Promise has exactly 3 states:

pending

Initial state — operation hasn't completed yet. Neither fulfilled nor rejected.

fulfilled

Operation completed successfully. The .then() callback runs with the result.

rejected

Operation failed. The .catch() callback runs with the error. Once a Promise settles (fulfilled or rejected), it cannot change state again — it's immutable.

JavaScript — Promises Complete
// ── CREATING A PROMISE ──
const myPromise = new Promise((resolve, reject) => {
  const success = true;

  setTimeout(() => {
    if (success) {
      resolve('Data loaded successfully!');  // fulfills the promise
    } else {
      reject(new Error('Failed to load data')); // rejects the promise
    }
  }, 1000);
});

// ── CONSUMING A PROMISE: .then() / .catch() / .finally() ──
myPromise
  .then(result => {
    console.log('Success:', result);
    return result.toUpperCase();  // return value passes to next .then()
  })
  .then(upperResult => {
    console.log('Chained:', upperResult); // .then() chains create a pipeline
  })
  .catch(error => {
    console.error('Error:', error.message);  // runs if ANY .then() in chain throws/rejects
  })
  .finally(() => {
    console.log('Done — runs regardless of success/failure');
  });

// ── REWRITING CALLBACK HELL with Promise CHAINING ──
function fetchUserPromise(id) {
  return new Promise(resolve => {
    setTimeout(() => resolve({ id, name: 'Poorv' }), 500);
  });
}
function fetchOrdersPromise(userId) {
  return new Promise(resolve => {
    setTimeout(() => resolve([{ id: 101, item: 'Laptop' }]), 500);
  });
}

fetchUserPromise(1)
  .then(user => fetchOrdersPromise(user.id))   // FLAT, not nested! ← the fix
  .then(orders => console.log(orders))
  .catch(err => console.error('One error handler for the whole chain!', err));

// ── PROMISIFYING old callback-based functions ──
function delay(ms) {
  return new Promise(resolve => setTimeout(resolve, ms));
}
delay(1000).then(() => console.log('1 second has passed'));

Async/Await
Syntactic sugar over Promises — write async code that reads like sync code

What async/await Really Is

async/await (ES2017) is built ON TOP of Promises — it's syntax sugar that makes Promise-based code look and read like synchronous code, without the .then() chains. Under the hood, it's still Promises — await pauses execution of the async function until the Promise settles.

Two rules to remember: (1) The async keyword makes a function always return a Promise. (2) await can ONLY be used inside an async function (or at the top level of modules) — it pauses that function until the awaited Promise resolves or rejects.
JavaScript — Async/Await Complete
// ── BASIC ASYNC FUNCTION ──
async function getData() {
  return 'Hello';
}
getData();  // returns a Promise that resolves to 'Hello' — NOT the string directly!
getData().then(result => console.log(result)); // 'Hello'

// ── CONVERTING the Promise chain to async/await (much more readable!) ──

// BEFORE (Promise chains):
function loadUserData() {
  return fetchUserPromise(1)
    .then(user => fetchOrdersPromise(user.id))
    .then(orders => {
      console.log(orders);
      return orders;
    })
    .catch(err => console.error(err));
}

// AFTER (async/await — reads top to bottom like sync code!):
async function loadUserDataAsync() {
  try {
    const user   = await fetchUserPromise(1);     // waits HERE until resolved
    const orders = await fetchOrdersPromise(user.id); // then waits HERE
    console.log(orders);
    return orders;
  } catch (err) {
    console.error(err);  // catches errors from ANY await above
  }
}

// ── SEQUENTIAL vs PARALLEL await — a critical performance distinction ──

// ❌ SEQUENTIAL (slow) — each await waits for the previous to finish
async function slowVersion() {
  const a = await fetchA();  // takes 1s
  const b = await fetchB();  // takes 1s — starts only AFTER fetchA finishes
  // Total time: ~2 seconds
}

// ✅ PARALLEL (fast) — start both at once, wait for both together
async function fastVersion() {
  const [a, b] = await Promise.all([fetchA(), fetchB()]);
  // Total time: ~1 second (both run simultaneously!)
}

// ── AWAIT IN LOOPS — common bug pattern ──
const ids = [1, 2, 3];

// ❌ Sequential, slow — one request after another
for (const id of ids) {
  const data = await fetchUserPromise(id);
  console.log(data);
}

// ✅ Parallel, fast — all requests fired at once
const allUsers = await Promise.all(ids.map(id => fetchUserPromise(id)));

// ── TOP-LEVEL AWAIT (ES2022, in modules) ──
// In a module file (type="module"), you can await without wrapping in async:
const config = await fetch('/config.json').then(r => r.json());

The Fetch API
Making real HTTP requests from JavaScript — connecting to backends

What is Fetch?

fetch() is the modern, built-in browser API for making HTTP requests. It returns a Promise that resolves to a Response object. This is exactly what you'll use to connect your frontend (React) to your backend (Node.js/Express) APIs.

Critical gotcha: fetch() only rejects on NETWORK failure (no internet, DNS error). It does NOT reject for HTTP error statuses like 404 or 500! You must manually check response.ok or response.status.
JavaScript — Fetch API Complete
// ── BASIC GET REQUEST ──
async function getUsers() {
  try {
    const response = await fetch('https://api.example.com/users');

    // ⚠️ fetch does NOT throw for 404/500 — must check manually!
    if (!response.ok) {
      throw new Error(`HTTP error: ${response.status}`);
    }

    const users = await response.json();  // parse JSON body
    return users;
  } catch (error) {
    console.error('Fetch failed:', error.message);
    throw error;  // re-throw so calling code can also handle it
  }
}

// ── POST REQUEST (sending data) ──
async function createUser(userData) {
  const response = await fetch('https://api.example.com/users', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'Authorization': `Bearer ${token}`  // auth token for protected routes
    },
    body: JSON.stringify(userData)
  });

  if (!response.ok) {
    const errorData = await response.json();
    throw new Error(errorData.message || 'Request failed');
  }

  return response.json();
}

// ── PUT / PATCH / DELETE ──
await fetch(`/api/users/${id}`, {
  method: 'PUT',    // full replace
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify(updatedUser)
});

await fetch(`/api/users/${id}`, {
  method: 'PATCH',  // partial update
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ name: 'New Name' })
});

await fetch(`/api/users/${id}`, { method: 'DELETE' });

// ── RESPONSE METHODS — different ways to read the body ──
response.json()    // parse as JSON (most common)
response.text()    // raw text
response.blob()    // binary data (images, files)
response.formData() // form-encoded data

// ── RESPONSE PROPERTIES ──
response.ok          // true if status 200-299
response.status      // 200, 404, 500, etc.
response.statusText  // 'OK', 'Not Found', etc.
response.headers     // Headers object
response.headers.get('content-type'); // read a specific header

// ── ABORTING A FETCH (timeout pattern) ──
async function fetchWithTimeout(url, timeoutMs = 5000) {
  const controller = new AbortController();
  const timeoutId = setTimeout(() => controller.abort(), timeoutMs);

  try {
    const response = await fetch(url, { signal: controller.signal });
    clearTimeout(timeoutId);
    return response;
  } catch (err) {
    if (err.name === 'AbortError') {
      throw new Error('Request timed out');
    }
    throw err;
  }
}

Error Handling
try/catch, custom errors, and graceful failure
JavaScript — Error Handling Complete
// ── try/catch/finally ──
try {
  const result = 10 / getDivisor();  // might throw
  console.log(result);
} catch (error) {
  console.error('Something went wrong:', error.message);
} finally {
  console.log('This always runs — cleanup code goes here');
}

// ── THROWING CUSTOM ERRORS ──
function validateAge(age) {
  if (typeof age !== 'number') {
    throw new TypeError('Age must be a number');
  }
  if (age < 0 || age > 120) {
    throw new RangeError('Age must be between 0 and 120');
  }
  return age;
}

try {
  validateAge(-5);
} catch (error) {
  if (error instanceof RangeError) {
    console.log('Range error:', error.message);
  } else if (error instanceof TypeError) {
    console.log('Type error:', error.message);
  }
}

// ── CUSTOM ERROR CLASSES (used in real backend projects) ──
class ApiError extends Error {
  constructor(message, statusCode) {
    super(message);
    this.name = 'ApiError';
    this.statusCode = statusCode;
  }
}

async function apiCall() {
  const response = await fetch('/api/data');
  if (!response.ok) {
    throw new ApiError('Failed to fetch', response.status);
  }
  return response.json();
}

try {
  await apiCall();
} catch (error) {
  if (error instanceof ApiError && error.statusCode === 401) {
    redirectToLogin();
  } else {
    showGenericError();
  }
}

// ── try/catch with async/await + Promise.all ──
async function loadDashboard() {
  try {
    const [user, orders, notifications] = await Promise.all([
      fetch('/api/user').then(r => r.json()),
      fetch('/api/orders').then(r => r.json()),
      fetch('/api/notifications').then(r => r.json())
    ]);
    return { user, orders, notifications };
  } catch (error) {
    // If ANY of the 3 requests fail, Promise.all rejects immediately
    console.error('Dashboard load failed:', error);
    throw error;
  }
}

JavaScript — Promise Combinators
// ── Promise.all() — waits for ALL, fails fast if ANY rejects ──
const [users, products, orders] = await Promise.all([
  fetch('/api/users').then(r => r.json()),
  fetch('/api/products').then(r => r.json()),
  fetch('/api/orders').then(r => r.json())
]);
// If ANY promise rejects, Promise.all immediately rejects with that error
// — even if the other 2 succeeded. Use when you need ALL results.

// ── Promise.allSettled() — waits for ALL, regardless of success/failure ──
const results = await Promise.allSettled([
  fetch('/api/users'),
  fetch('/api/maybe-broken-endpoint'),
  fetch('/api/products')
]);
// Returns: [
//   { status: 'fulfilled', value: Response },
//   { status: 'rejected',  reason: Error },
//   { status: 'fulfilled', value: Response }
// ]
// Never throws — gives you ALL outcomes, even failures.
// Use when you want partial results (e.g. show what loaded, skip what failed)

results.forEach(result => {
  if (result.status === 'fulfilled') {
    console.log('Success:', result.value);
  } else {
    console.log('Failed:', result.reason);
  }
});

// ── Promise.race() — settles as soon as the FIRST one settles (win or lose) ──
const result = await Promise.race([
  fetch('/api/data'),
  new Promise((_, reject) => setTimeout(() => reject('Timeout!'), 5000))
]);
// Classic pattern: race a real request against a timeout promise

// ── Promise.any() — settles as soon as the FIRST one FULFILLS (ignores rejections) ──
const fastestSuccess = await Promise.any([
  fetch('https://mirror1.example.com/data'),
  fetch('https://mirror2.example.com/data'),
  fetch('https://mirror3.example.com/data')
]);
// Only rejects if ALL promises reject. Useful for redundant/mirror servers.

Real Project: Data Fetching Pattern
The exact pattern used in React apps to load and display API data
🏢 Real World

This pattern is used in literally every React app

Loading state, error state, success state — this three-state pattern around async data fetching is universal. You'll write this exact structure inside useEffect in React on Day 9.

JavaScript — Complete Data Fetching Module
// ── A reusable API client wrapper ──
class ApiClient {
  constructor(baseUrl) {
    this.baseUrl = baseUrl;
  }

  async request(endpoint, options = {}) {
    const url = `${this.baseUrl}${endpoint}`;

    try {
      const response = await fetch(url, {
        headers: {
          'Content-Type': 'application/json',
          ...options.headers
        },
        ...options
      });

      if (!response.ok) {
        const errorBody = await response.json().catch(() => ({}));
        throw new Error(errorBody.message || `HTTP ${response.status}`);
      }

      return await response.json();
    } catch (error) {
      console.error(`API request failed: ${endpoint}`, error);
      throw error;
    }
  }

  get(endpoint)        { return this.request(endpoint); }
  post(endpoint, data)  { return this.request(endpoint, { method: 'POST', body: JSON.stringify(data) }); }
  put(endpoint, data)   { return this.request(endpoint, { method: 'PUT', body: JSON.stringify(data) }); }
  delete(endpoint)      { return this.request(endpoint, { method: 'DELETE' }); }
}

const api = new ApiClient('https://api.example.com');

// ── THREE-STATE PATTERN (loading / error / success) ──
async function loadProductsPage() {
  const state = { loading: true, error: null, data: null };
  renderUI(state);  // shows a loading spinner

  try {
    const products = await api.get('/products');
    state.loading = false;
    state.data = products;
  } catch (error) {
    state.loading = false;
    state.error = error.message;
  }

  renderUI(state); // shows either the data or an error message
}

// ── USING THE CLIENT FOR A FULL FEATURE ──
async function checkoutFlow(cartItems) {
  try {
    // Step 1: Validate stock for all items in parallel
    const stockChecks = await Promise.all(
      cartItems.map(item => api.get(`/products/${item.id}/stock`))
    );

    const allInStock = stockChecks.every(check => check.available);
    if (!allInStock) {
      throw new Error('Some items are out of stock');
    }

    // Step 2: Create order (sequential — depends on stock check passing)
    const order = await api.post('/orders', { items: cartItems });

    // Step 3: Process payment
    const payment = await api.post('/payments', { orderId: order.id });

    return { success: true, order, payment };
  } catch (error) {
    return { success: false, error: error.message };
  }
}

Placement Interview Q&A
Top async JavaScript questions — these are guaranteed to come up
  • Q1. Explain the event loop and why setTimeout(fn, 0) doesn't run immediately.
    JavaScript has a single call stack. Async operations (timers, network calls) are handed off to the browser/Node APIs, and when they complete, their callback goes into a queue (microtask queue for Promises, macrotask/callback queue for setTimeout). The event loop only processes these queues when the call stack is completely empty. So setTimeout(fn, 0) doesn't run "in 0ms" — it gets queued, and only runs after ALL currently executing synchronous code finishes, AND after the microtask queue is drained. This is why console.log order can surprise people: sync code always wins first, then microtasks (Promises), then macrotasks (setTimeout).
  • Q2. What is a Promise and what are its three states?
    A Promise is an object representing the eventual result of an asynchronous operation. It has three states: pending (initial, operation not complete), fulfilled (operation succeeded, value available via .then()), and rejected (operation failed, error available via .catch()). Once a Promise settles (fulfilled or rejected), it's immutable — it cannot change state again. Promises solved "callback hell" by allowing chainable .then() calls instead of deeply nested callbacks.
  • Q3. What is async/await and how does it relate to Promises?
    async/await is syntactic sugar built on top of Promises that makes asynchronous code look synchronous and easier to read. An async function always returns a Promise (even if you just `return value`, it gets wrapped). The await keyword can only be used inside async functions — it pauses execution of that function until the awaited Promise settles, then either returns the resolved value or throws the rejection as a catchable error. Under the hood, it's still all Promises — async/await doesn't replace them, it's a cleaner way to write Promise-based code.
  • Q4. What is the difference between Promise.all and Promise.allSettled?
    Promise.all waits for all promises to fulfill, but if ANY single one rejects, it immediately rejects with that error — you lose the results of the other promises even if they succeeded. Use it when you need ALL results to proceed (e.g., loading 3 pieces of data all required for a page). Promise.allSettled waits for ALL promises to settle regardless of outcome, and returns an array of {status, value/reason} objects for each — it never rejects. Use it when partial success is acceptable, like loading multiple independent widgets where one failing shouldn't break the others.
  • Q5. Why doesn't fetch() throw an error for a 404 or 500 response?
    fetch() only rejects its Promise on network-level failures — no internet connection, DNS failure, CORS block. It does NOT treat HTTP error status codes (404, 401, 500) as failures because, from a networking perspective, the request DID succeed — the server simply responded with an error status. This means you must manually check response.ok (true for status 200-299) or response.status and throw your own Error if needed. This is one of the most common bugs junior developers make — assuming fetch().catch() will catch all API errors.
  • Q6. How would you run 5 independent API calls in parallel and wait for all of them?
    Use Promise.all() with an array of fetch calls (or async function calls): const results = await Promise.all([fetch(url1), fetch(url2), fetch(url3), fetch(url4), fetch(url5)]). This starts all 5 requests at the same time rather than waiting for each one sequentially with separate await statements, which would take 5x as long if each request takes 1 second. If you need partial results even when some fail, use Promise.allSettled() instead.
  • Q7. What happens if you forget to use await before an async function call?
    You get the Promise object itself, not the resolved value. For example, if getData() is async and returns 'Hello', calling const result = getData() (without await) makes result a pending Promise object, not the string 'Hello'. You'd need result.then(value => ...) or await getData() to get the actual string. This is a very common bug — forgetting await leads to code that tries to use a Promise object as if it were the final value (e.g., calling .map() on a Promise instead of the array it resolves to, which throws a TypeError).
  • Q8. How do you handle errors in async/await code, and why is try/catch important?
    Wrap await calls in a try/catch block — if the awaited Promise rejects, the rejection becomes a regular JavaScript error that the catch block can handle, just like synchronous throw/catch. Without try/catch, an unhandled rejection in an async function will cause the function's returned Promise to reject, and if nothing catches that downstream, you get an "Unhandled Promise Rejection" warning (and potential app crashes in Node.js). Always wrap network calls, JSON parsing, and other failure-prone operations in try/catch, and consider a finally block for cleanup (like hiding a loading spinner) that should run regardless of success or failure.
